# Demonstrating a Future for MLIR-native DSL Compilers on a NumPy-like Example 

## KARL F. A. FRIEBEL, TU Dresden, Germany 

## JASCHA A. OHLMANN, TU Dresden, Germany 

## JERONIMO CASTRILLON, TU Dresden, Germany 

Compilers for general-purpose languages have been shown to be at a disadvantage when it comes to specialized application domains as opposed to their Domain-Specific Language (DSL) counterparts. However, the field of DSL compilers features little consolidation in terms of compiler frameworks and adjacent software ecosystems. As a result, considerable work is duplicated, lost to maintenance issues, or remains undiscovered, and most DSLs are never considered “production-ready”. One notable development is the introduction of the Multi-Level Intermediate Representation (MLIR), which promises a similar impact on DSL compilers as LLVM had on general-purpose tooling. In this work, we present a NumPy -like DSL made for offloading numeric tensor kernels that is entirely MLIR-native. In a first for open-source, it implements all frontend actions and semantic analyses directly within MLIR. Most notably, this is made possible by our new dialect-agnostic MLIR type checker, created for the future of DSLs in MLIR. We implement a simple, yet effective, parallel-first lowering scheme that connects our language to another MLIR dataflow dialect for seamless offloading. We show that our approach performs well in real-world use cases from the domain of weather modeling and Computational Fluid Dynamics (CFD) in Fortran. Additional Key Words and Phrases: MLIR, DSL, type checking, tensor algebra, offloading, compute kernels 

1 Introduction 

A DSL provides a concise and idiomatic representation for a specific class of problems. By directly borrowing domain terminology, users are freed from obtuse coding patterns that are barriers to entry and readability. With their limited scope, DSL compilers can sacrifice generality for simplicity and still meet the same quality of result. Unsurprisingly, there is a sprawl of DSLs in the published research landscape. However, few have been adopted widely enough to leave academic circles. A proposed solution is to build DSL compilers in MLIR, an extensible Intermediate Representation (IR) framework. This allows them to interoperate, letting DSL authors benefit directly from the work of their peers. MLIR also reduces the dependency surface of the DSL compiler, letting the wider MLIR user base help maintain it. Although advertised as “batteries included”, using MLIR properly (i.e., reusably) still requires lots of experience. This paper presents two contributions to the field of DSL compilers: 

• A blueprint for MLIR-native DSLs (Section 3), i.e., deeply embedding DSLs in MLIR to reduce bespoke infras-tructure and increase reusability. 

• A minimal and safe extension of MLIR (Section 4) to introduce subtyping and a reusable deductive type checker, used by our DSL frontend. As a demonstration, we designed and implemented a NumPy -like DSL in MLIR, up to and including Abstract Syntax Tree (AST) level and diagnostics. We implemented a parallel-first, pipeline-later lowering (Section 5) to dataflow circuits in MLIR [ 3 , 24 ] for targeting heterogeneous execution platforms [ 23 ]. 2.4 Compiling DSLs with MLIR 

MLIR [ 13 ] is an IR often presented as ideal for DSLs. It is extensible, meaning that users can add their own semantics via dialects. Dialects are sets of operations (ops), attributes and types that may appear in MLIR’s fixed structure. There is also a set of “core dialects”, which are abstractions maintained together by the community. One of the essential core dialects is the llvm dialect that maps to LLVM-IR. It acts as a common back-end to the whole MLIR ecosystem. Creating a DSL in MLIR can be as straightforward as adding a new dialect and a transitive conversion (i.e., lowering) to llvm .MLIR is also a compiler framework, providing a pass infrastructure and diagnostics engine. Creating a compiler using MLIR is similar to LLVM: the compiler populates an IR module, runs some pass pipeline on it, and then consumes the result to emit some artifact. However, MLIR does not feature complete end-to-end compilers upstream. Along with other recurring frontend matters, e.g. type checking, a feature-rich DSL frontend presently still requires a substantial codebase that is not native to MLIR. Dialects are the fundamental unit of reuse in MLIR and separate concerns under some completeness requirement. For complex languages like C++, frontends may require multiple dialects for elaboration tasks before reaching executable semantics and code generation. The ClangIR [ 1] project is creating such an MLIR stack for the C family of languages, but does not encompass the full AST 8.

3.3 Type Checking

Only well-typed programs are semantically valid, and the frontend must check that such an assignment exists, or reject the program. In status quo MLIR, type checking happens during IR verification, where each op typically checks its operand types. 3.4 Benefits and Drawbacks 

DSLs that follow the same blueprint we used for EKL benefit from fast and safe iterative design. Little logic needs to go in the parser, and semantic analyses can be reused at the MLIR level. Additionally, there is no code duplication when implementing the type system or syntactic sugar. Our extensions to the infrastructure remove the hurdle on entry to MLIR. MLIR-native DSL compilers 9On the other hand, MLIR is a complex framework that requires more boilerplate code than a bespoke IR. With an emphasis on integration and maintenance, however, we expect this to be amortized by reuse. Complex languages may also require a hierarchy of frontend dialects (see [1]), which may lead to issues with backtracing. 

4 Subtyping & Type Checking in MLIR 

This section describes our open-source 10 extension to the MLIR infrastructure to support type systems. It introduces the concept of subtyping to MLIR, via a dialect interface. This relation subsumes the value substitutability rule, minimally changing the core of MLIR. To allow for efficient in-place IR mutation, we extend substitutability to transmutability. We implemented a dialect-agnostic type checking pass, which runs a fix-point algorithm on op typing rules advertised by a new op interface. 

4.1 Status-quo typing in the MLIR SSA IR 

Every value in MLIR requires a type. Though MLIR has some built-in types, its core remains unopinionated. Dialects can introduce new types, freely assign semantics to them, and even reinterpret foreign types in their semantics. An MLIR type is simply a family of compile-time attributes with static lifetime and a notion of definitional equality. There have been previous attempts at embedding richer type systems into MLIR, with features such as type abstractions. For example, CIRCT module parameters are upstream type abstractions based on a simple but unsafe 11 

workaround. Safe approaches embed type expressions into MLIR by creating self-contained reasoning frameworks, which may bypass MLIR types completely. This has been comprehensively demonstrated by Rise [ 15 ] and MimIR [ 14 ]. However, these embeddings must remain closed, i.e., foreign expressions and types must be encapsulated. All such embeddings lose the desired orthogonality of MLIR dialects, because they introduce composition hazards. The alternative is to modify the MLIR core to add first-class support for rich type systems. For features like dependent types, this requires sweeping changes to the ABI, API, runtime and memory consumption of MLIR that its users care about and maintainers defend. The majority of MLIR users does not desire for it to become a theorem prover itself. Such a change would run a high risk of fracturing the project. Our extension is informed by the de-facto use of MLIR and its development philosophy. The implementation described in this section concedes that any enrichment of types in MLIR is limited by compatibility issues and non-functional constraints on MLIR. Instead, we propose a reasonable patch that minimally modifies the spec and existing code of MLIR in a way that preserves the status quo while providing useful reasoning power and allowing gradual adoption. 

4.2 The DialectTypeSystemInterface 

Values in MLIR are considered substitutable when their types are definitionally equal. Although this is an implicit contract that can be violated by op verifiers, it is one of the few invariants that is opinionated in the core of MLIR. Subtyping relaxes this constraint throughout the core, touching on three core interfaces. To introduce an extensible subtype relation to MLIR core, a new component is added to it such that: 

• It decides value substitutability. 

• It is extensible via dialect loading. 

• The load order of independent dialects can not matter. 

• Dialects can disagree on type semantics.

• It does not change the behavior of existing compilers. This is realized using the DialectTypeSystemInterface that is attached to dialects at load time. Attaching to dialects rather than ops reduces the performance impact, assuming that all ops in a dialect agree on the same subtype relation. This interface is an opt-in mechanism that allows querying the subtype relation and performing promotions. It is also a starting point for future extensions. The status quo is preserved in the EqualityTypeSystem default implementation. All implementations must uphold reflexivity and transitivity, so that performance degradation is avoided in compilers that do not use this new feature. Modifications to the core interfaces ensure that opted-in dialects get the expected behavior when dealing with control flow edges in their own IR. For performance reasons, we extend substitutability with transmutability. During deduction, a common scenario is that a more concrete type is found for 𝑣 that satisfies all of its current users. Ideally, the type of the value 𝑣 is updated in-place, avoiding costly IR mutations. Using our interface, we can now determine when this transmutation is legal. Transmutability is more involved than evaluating the subtype relation, because of BlockArgument s. These are values defined not by operations, but blocks contained within them. In practice, we follow the control flow interfaces in addition to SSA dataflow. Since operations may have attributes synced to block arguments, e.g., function signatures, we allow customizing transmutation. As before, transmutation requires opt-in, to preserve backward compatibility. 

4.3 The TypeCheckOpInterface 

MLIR core features an InferTypeOpInterface , which is the existing type deduction mechanism. It is specially privileged by MLIR, as it is known to the Operation Definition Specification (ODS) and its parsers. Primarily, it can be used to infer the result types of a yet unconstructed op from constructor arguments. This allows omitting the types of op results in MLIR’s textual format. It introduces a per op verifier for status quo type checking. This interface is unsuitable for us, because it does not capture the concept of typing rules adequately. And, while a typing rule subsumes it, its privileged status makes it somewhat orthogonal. It serves a specific purpose in the context of ODS and its automatically generated C++ template builders. We conclude that ops with syntactically inferable types should continue to use this interface. We introduce the TypeCheckOpInterface to advertise typing rules. Ops must override the typeCheck method, which evaluates the typing rule. Implementations can also override a set of methods that control transmutation of results and block arguments. The interface also adds a per op verifier to work with status quo type checking. This feature is preserved by checking the rule locally against the current state of the IR, avoiding further code duplication. The typeCheck method takes an AbstractTypeChecker and returns an optional Contradiction . The type checker is used to observe the current context and make deductions, i.e., add new constraints. The contradiction result is used to implement rich diagnostics that explain when type checking gets stuck. The typing rule can be invoked multiple times during checking, whenever the context of the operation changes. We use an adaptor to extend AbstractTypeChecker with dialect-specific functions, like broadcastAndPromote .

When the fix-point is reached, the pass atomically applies all deductions to the IR. By sampling the most restrictive bound, a type is found for each value, which is materialized in the IR via transmutation and/or casting, depending on the users. Using our infrastructure, the IR is able to remain in a verified state throughout, which is important when it is used as an AST. The fix-point algorithm keeps a priority queue of rules that need to be (re-)applied. Since these rules are given by ops, the queue is ordered in canonical lexical order of the IR, which also helps reduce the number of iterations for deductions along SSA edges. Each iteration, a rule is dequeued and applied. Assuming no contradictions are found, the new deductions are recorded into the state. Deductions that refine the type of a value cause its users to be invalidated. Because these SSA values form a dataflow Directed Acyclic Graph (DAG), this process will terminate. This termination guarantee is also preserved when rules explicitly invalidate child ops, as MLIR is also a DAG. However, to accommodate idiomatic structural MLIR patterns, some parent ops also need to be invalidated. Consider the following idiomatic MLIR example, where %i is a capture operand and %j its receiving block argument.    

> %r = container (%j = %i) {
> yield %j }

During inference, container will deduce the type of %j to be the type of %i . This causes yield ’s rule to be applied, but that op does not own the result %r . As a result, yield must propagate the invalidation to its parent, which can constrain %r . In this example, no infinite loop occurs, because the types of %j and %r are independent. We hope that, in future, safe typing rule code is generated declaratively by ODS. 

5 The EKL Middle-end 

In Section 2.3, we introduced the parallel-first, pipeline-later strategy for compiling tensor programs. This strategy assumes that the optimal solution can be found from a maximum-parallel normalization form by gradually removing parallelism guided by the targets resources, e.g., hardware vector size. Lowering EKL programs to executable target code follows this strategy, preserving target independence for as long as possible. 

5.1 Program Normalization 

The first step is to normalize the program into a map-reduce form. This is achieved using the inherently parallel generator expressions (cf. Fig. 2), a core design feature of EKL. The assoc generator produces an array by associating tuples of an index space with values returned by a child functor. The zip generator produces an array by combining the values of its operands at matching index tuples using a child functor. The reduce generator produces a scalar from an array by applying a reduction functor. The first three stages of our pipeline establish this parallelized normal form. Stage 1 is a cursory operator simplification, which rewrites compound expressions within ekl . Stage 2 resolves the debt incurred by our single-step type checking (cf. Section 3.3), making implicit casts explicit. Stage 3 rewrites array expressions, resulting in a mix of the embarassingly parallel assoc and reduce generators through aggressive inlining and code motion. Examples for stage 2 include materializing broadcasts and type promotions. Another example are NumPy ellipses ... 

subscripts, which expand to zero or more identity indexers : as needed. Single-step type checking has left us with other such implicit expansions in subscript expressions, which are made explicit here. 5.3 Operator Implementation 

We provide an operator implementation lowering via the upstream MLIR dialects tensor , linalg , arith and affine ,which is enough to enable code generation via LLVM. For more control over pipelining the operators, we use a pass that lifts all dataflow dependencies into the dfg-mlir dialect [ 3 ]. This can also be used to offload stages to heterogeneous compute devices, such as FPGAs via Bambu [8] and Olympus [23]. While the pipeline is target-agnostic up until this point, it must specialize for the desired hardware target now. To allow reuse of existing MLIR back-ends, computations must be exposed so that they can be mapped 1 : 1 onto target instructions. Some program constructs are not yet efficiently mapped to CPU targets, as we initially only targeted FPGAs. In particular, the FPGA target trades speed for area when parallel (re-)computations are combined with muxers, where the CPU prefers branches. For the CPU, we currently apply a simple producer-consumer fusion to minimize reuse distance. Introducing a single compiler flag, we are able to automatically extract parallel work share regions for the OpenMP runtime. This back-end can also be used to implement and test the dataflow network for FPGA offloading locally. However, the current state of MLIR bufferization means that the quality of result is still poor for CPU execution. We validated the results of the kernels against the baseline, and measured their latencies. Table 2 gives the baseline under gfortran and ours under eklc . The “seq.” and “par.” columns show single-thread and 8-thread OpenMP execution respectively. Time measurements are provided with 3 significant digits, and speed-ups with 2 significant digits. Speed-ups less than ×1 indicate an increase in latency. For specific applications, our naive approach turned out to be competitive with the expert-optimized code. While further target specialization in eklc is necessary to address the other kernels, we also identified key deficiencies in the MLIR back-ends reused by us. 

6.1 Experiment Setup 

Evaluation of the taumol_sw kernel was performed using a full WRF predicition run, the configuration and input data of which was provided by our partners. In the commonly used build configuration dmpar of WRF that we used, the kernel executes single-threaded. Instead, the workload is spatially distributed across threads. Our kernel is dynamically linked into WRF as an open-source 12 plugin, allowing the baseline to be swapped in and out. To easily reproduce the experiment, also on FPGA, we derived controlled experiment conditions from these runs. Since the kernel exhibits data-dependent timing, we recorded realistic replay data from the original simulation runs. We then replay this data to the kernel using a test adapter that calls our plugin library. For taumol_sw , we averaged the latency of 1 million single-column runs, which are the smallest unit of work WRF can distribute. The HiSPEET kernels were tested using a Fortran application that directly integrates the DSL. The baseline HiS-PEET implementation also has OpenMP pragmas, allowing a fair comparison with our OpenMP back-end. For the HiSPEET kernels, we averaged the latency of 50 runs on 8192 elements each, which exhibit per-element parallelism and vectorization opportunities.

6.3 Parallel Performance 

Our OpenMP back-end uses existing MLIR infrastructure to emit work sharing and reduction loops. This requires no modifications to the DSL code by the user, but relies heavily on the optimization capabilities of the middle-end dialects we reuse. The flang Fortran compiler uses the same MLIR OpenMP facilities, but not the other dialects we use. Expecting to be limited by the single-thread performance of eklc , we still see a substantial speed-down for all our kernels, compared to the OpenMP baseline. Bufferization, i.e., the assignment of memory buffers to tensor values, was found to be the main cause of this. Bufferization is implemented by a piece of MLIR infrastructure that we reuse. In the single-threaded case, currently existing analyses and optimizations are able to, e.g., eliminate spurious copies and allocations. When lowering to OpenMP, the parallel nature of loops requires a different bufferization scheme, which is not established by existing passes. In addition, the dynamic allocations de-facto serialize some parallel regions. To generate efficient parallel implementations, eklc requires an improved bufferization mechanism. Other deficiencies include optimization hazards that are created around outlined OpenMP regions. CPU-specific lowerings, as mentioned above, also become more relevant in the parallel case. Specifically, we identified the need for immutable references to intermediates resulting from certain subscripts that can be (re-)used in parallel. 

6.4 FPGA Offloading 

While WRF’s SPMD use of the CKD kernel is not particularly amenable to offloading due to WRF’s software architecture, we evaluated this for an FPGA target. We used High-Level Synthesis (HLS) to implement our operators, using Bambu [ 8]and CIRCT 13 , as both allow us to use MLIR as an input. The dfg-mlir project allows us to model our kernels as Khan Process Networks (KPNs) that replace memory buffers with streaming channels. Etna [ 24 ] is able to consume this representation and generate a memory-attached accelerator, complete with host driver code, our plugin can link against. This flow requires the process nodes to be synthesized with AXI stream interfaces first.

7 Conclusion 

While the promise of MLIR lies in reusability, practical DSL implementations still require a lot of development effort and careful design to achieve it. We demonstrated that feature-rich DSL compilers can be implemented in MLIR without resorting to bespoke or legacy implementations. In the process, we showed that specific low-friction MLIR extensions, such as lexical locations and type checking, greatly simplify front-end creation and promote reuse. We created an open-source MLIR patch that adds subtyping and dialect-agnostic type checking as a first step towards MLIR-native front-ends. By applying it all to a rather complex sample application, we showed that the promised advantages of MLIR are attainable even in its still limited ecosystem. In fact, this reuse of established techniques and abstractions was uniquely made possible by MLIR alone. However, we also faced some limitations that the current machine learning focused MLIR ecosystem poses to other uses. We intend to use eklc to identify and address more such issues to improve MLIR.
