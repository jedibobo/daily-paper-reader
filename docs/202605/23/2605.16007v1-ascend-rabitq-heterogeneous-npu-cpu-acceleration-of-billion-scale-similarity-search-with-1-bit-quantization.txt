## Abstract 

Vector similarity search is a critical component of modern AI systems, but traditional CPU-based implementations face fun-damental scalability bottlenecks for billion-scale corpora due to prohibitive computational overhead and memory bandwidth lim-itations. While Neural Processing Units (NPUs) offer orders-of-magnitude higher compute density, existing CPU/GPU-optimized 1-bit RaBitQ quantization implementations cannot be directly ported to NPU architectures due to fundamental hardware mismatches, and homogeneous design paradigms struggle to simultaneously balance accuracy, memory footprint, and performance. This paper presents Ascend-RaBitQ, the first heterogeneous NPU-CPU optimized IVF-RaBitQ system for billion-scale vector search, built on the core insight that decoupling coarse ranking (NPU) from fine ranking (CPU) allows each stage to leverage its optimal hardware, breaking the long-standing accuracy-memory-performance trade-off. We propose a three-stage heterogeneous pipeline comprising AI Core-accelerated coarse ranking on 1-bit quantized vectors, on-device AI CPU Top-𝑘 processing, and host CPU fine re-ranking on full-precision vectors. We introduce four NPU architecture-native optimizations: fused AIC-AIV operators for parallel distance computation, computation flow restructur-ing to exploit rotation orthogonality, fine-grained index block-level load balancing that breaks query boundaries, and intra-NPU pipeline parallelism between AI Core and AI CPU to mask Top-𝑘 

latency. Evaluation on standard datasets shows that Ascend-RaBitQ achieves 3.0 × to 62.8 × faster index construction than the CPU baseline, up to 4.6 × throughput improvement over the fastest CPU IVF-RaBitQ implementation, and over 100 × over the mathemati-cally equivalent CPU baseline, while demonstrating encouraging scalability on distributed multi-NPU systems.

## 1 Introduction 

Vector similarity search has become a critical infrastructure compo-nent in modern AI applications, powering retrieval-augmented generation (RAG) [ 22 , 25 , 34 , 36 ], search and recommendation systems [ 5, 24 , 29 , 35 ], and computer verison [ 9, 10 , 26 , 41 ]. As large language models (LLMs) and agentic AI systems continue to evolve, the demand for efficient, high-throughput vector retrieval 

> ∗Corresponding authors.

has grown exponentially, with industry reports indicating vector databases must now handle billion-scale corpora while maintaining millisecond-level latency and cost-effectiveness [ 1, 23 , 38 ]. Tradi-tional vector search systems predominantly run on CPU clusters, employing indexing structures such as inverted file systems (IVF) and product quantization (PQ) to reduce search space [ 4, 13 , 20 ], but face fundamental scalability challenges: (1) computational bottlenecks —CPU-based distance computations become prohibi-tively expensive as corpus size grows [ 23 ]; (2) memory bandwidth limitations —fetching high-dimensional vectors from main memory dominates latency, forming a well-documented "memory wall" for vector retrieval workloads [23]. Neural Processing Units (NPUs) have emerged as transformative accelerators for deep learning workloads, delivering orders-of-magnitude improvements in throughput compared to CPUs [ 30 ]. This exceptional compute density has driven its widespread adop-tion in data centers [ 7, 16 , 45 ], while making it highly suitable for data-intensive workloads such as vector similarity search. However, realizing this potential for vector retrieval introduces two funda-mental challenges. First, NPUs feature limited on-device memory capacity (typically 32-64 GB HBM) compared to CPU clusters with terabyte-scale DRAM pools, making it infeasible to store billion-scale high-dimensional vector indices entirely on-device. Scaling capacity by adopting multiple NPUs is prohibitively expensive, while simply offloading computation to NPUs incurs costly host-device data transfers that erase acceleration gains. Second, while aggressive quantization can reduce index size to fit NPU memory, existing schemes such as PQ incur severe accuracy degradation at extreme compression ratios, forcing a trade-off between memory footprint and retrieval quality. Recent advances in quantization theory offer a promising solu-tion: Randomized Bit Quantization (RaBitQ) [ 12 ] is a 1-bit quan-tization scheme with two core characteristics—formal theoretical error guarantees proven to be asymptotically optimal, and 32 ×

memory compression compared to FP32 vectors. At the same compression ratio, RaBitQ achieves substantially higher accuracy than PQ. This extreme compression provides a technically viable path to address the NPU memory constraint, reducing billion-scale vector indices from terabytes to tens of gigabytes so they fit entirely within a typical Ascend NPU’s 64 GB HBM, eliminating costly host-device data transfers. Existing RaBitQ implementations have been well optimized for both CPU and GPU platforms, with CPU-based designs leveraging specific instructions for fast bit-wise operations [ 12 ] and GPU implementations leveraging CUDA

warp-level primitives to achieve 2.7× higher throughput than IVF-PQ [ 12 , 37 ]. However, these CPU/GPU RaBitQ implementations cannot be directly adapted to NPU platforms for two critical reasons. First, both CPU and GPU RaBitQ implementations rely on architectural features that differ fundamentally from Ascend NPU. CPU-based RaBitQ depends on SIMD extensions for population count and bitwise operations. GPU-based RaBitQ leverages CUDA’s unified memory model, warp-level primitives, and thousands of CUDA cores. Ascend NPU’s Da Vinci architecture adopts a funda-mentally different design philosophy optimized for AI workloads: it features a heterogeneous compute fabric comprising AI Cores (each containing either a Cube Unit for matrix operations or a Vector Unit for element-wise operations) alongside dedicated AI CPUs for general-purpose scalar computation and task scheduling [ 30 ]. Unlike GPU’s hardware-managed L1/L2 cache hierarchy, Ascend employs a software-managed Unified Buffer (UB) and L1 Buffer for explicit memory orchestration [ 17 ], and its static dataflow execu-tion model requires careful algorithm redesign for non-standard operations like bitwise quantization. These architectural differences mean that existing CPU/GPU RaBitQ implementations cannot be directly ported—instead, we must embrace NPU’s heterogeneous design through algorithm-hardware co-optimization to unlock its performance potential for vector search. Second, and more fundamentally, homogeneous architec-tures—whether CPU-only, GPU-only, or NPU-only—fail to balance accuracy, memory footprint, and performance simultaneously. Pure CPU implementations struggle to meet throughput requirements for billion-scale datasets. Pure GPU implementations face a critical trade-off: 1-bit RaBitQ suffers from lower recall, while higher-bit RaBitQ variants [ 11 , 37 ] quickly exceed limited GPU memory capacity. Similarly, a pure NPU implementation, despite RaBitQ’s 

32 × compression fitting the quantized index in HBM, faces an inherent accuracy-performance conflict: 1-bit coarse ranking alone cannot deliver production-grade recall without refinement, but performing full-precision refinement on the NPU would require gathering raw vectors from CPU DRAM and transferring them across the host-device boundary—a data movement overhead that erases the NPU’s acceleration gains. No homogeneous architecture can simultaneously deliver high accuracy, high throughput, and manageable cost under practical deployment constraints. This leads to our core insight: a heterogeneous co-designed ap-proach —combining NPU acceleration for fast coarse ranking, CPU processing for high-accuracy fine ranking, and joint algorithm-hardware optimization—can break through the long-standing accuracy-memory-performance trade-off faced by homogeneous designs. This insight drives the design of Ascend-RaBitQ , the first NPU-accelerated RaBitQ library purpose-built for commercial datacenter NPU platforms. This paper presents the design, implementation, and evaluation of Ascend-RaBitQ. Our key contributions are: 

Heterogeneous Three-Stage NPU-CPU Collaborative Pipeline. We propose the first heterogeneous IVF-RaBitQ pipeline that decom-poses query processing across three hardware stages: AI Core for high-throughput 1-bit coarse distance computation, on-device AI CPU for low-latency Top-𝑘 selection, and host CPU for high-accuracy fine re-ranking on full-precision vectors. This design breaks the accuracy-memory-performance trilemma by mapping each stage to its optimal hardware. We implement pipeline par-allelism between AI Core and AI CPU to mask Top-𝑘 latency within the NPU, and offload re-ranking to host CPU, where the raw vectors are already resident, eliminating cross-device data movement overhead. Controlled ablation experiments show that the complete pipeline delivers a 23.1 × cumulative speedup over the naive baseline. Furthermore, the system achieves up to 4.6 × higher throughput than the fastest CPU IVF-RaBitQ implementation, over 100 × over the mathematically equivalent CPU 1-bit baseline, and up to 62.8 × faster index construction. 

NPU Architecture-Native RaBitQ Optimizations. We present the first RaBitQ implementation fully optimized for the Da Vinci heterogeneous compute architecture. Three key optimizations exploit its unique hardware characteristics: (1) a fused AIC-AIV operator that parallelizes matrix multiplication and vector norm calculation across the Cube and Vector units; (2) computation flow restructuring that exploits rotation orthogonality to overlap rotation with norm precomputation; (3) adaptation of 1-bit distance computation to the CANN native gather operator for FastScan-based computation, which achieves a 15.2 × cumulative speedup over the baseline select approach. These optimizations collectively unlock the NPU’s heterogeneous compute potential for RaBitQ-based vector search. 

Efficient Scheduling and Resource Orchestration. We introduce two scheduling techniques tailored for the NPU’s software-managed memory hierarchy: (1) fine-grained index block-level load balancing that breaks query boundaries, achieving a 6.3 × speedup; (2) IVF-aware Top-𝑘 pruning that prioritizes closer clusters, which combined with intra-NPU AI Core-AI CPU pipeline parallelism masks nearly all Top-𝑘 selection latency. Together, they eliminate both computation and memory access bottlenecks in the retrieval pipeline. 

Production Deployment and Distributed Scaling. We validate Ascend-RaBitQ in real production systems. We design native dis-tributed scaling for multi-NPU clusters, demonstrating up to 3 × end-to-end throughput speedup on 8 NPUs with encouraging distance computation scalability. Ascend-RaBitQ is open-sourced as part of the Ascend IndexSDK [6]. The remainder of this paper is organized as follows: Section 2 provides preliminaries on vector search and RaBitQ quantization. Section 3 details the NPU IVF-RaBitQ design and implementation. Section 4 presents our experimental evaluation. Section 5 discusses related work. Finally, Section 6 concludes the paper.

## 2.1 Approximate Nearest Neighbor Search and IVF Index 

High-dimensional vector similarity search is a core primitive for modern AI systems including Agentic AI with industrial deploy-ments now routinely supporting billion- to trillion-scale vector corpora [23, 27]. 

2.1.1 ANN Problem Definition. Given a vector corpus X =

{x1, x2, . . . , x𝑁 } where each x𝑖 ∈ R𝐷 , and a query vector q ∈ R𝐷 , the exact nearest neighbor search problem aims to find the vector x∗ ∈X that minimizes a distance metric 𝑑 (q, x∗) (typically Euclidean distance or inner product). Exact search has O( 𝑁 𝐷 ) complexity per query, which is prohibitively expensive for billion-scale workloads. Approximate Nearest Neighbor (ANN) search trades minor accuracy loss for orders-of-magnitude performance improvements, and is the standard approach in production deployments. 

2.1.2 IVF Index Structure. The Inverted File (IVF) index [ 20 ] is the de facto standard for billion-scale ANN production deployments. It reduces the search space via vector clustering and consists of two phases: 

Index Construction. Run 𝐾 -Means clustering on a training subset of the corpus to generate nList cluster centroids, each defining an inverted list. For each base vector x𝑖 ∈ X , assign it to the inverted list of its nearest centroid. 

Query Processing. For an input query q, compute its distance to all nList centroids, select the top nProbe nearest clusters as the search scope, and evaluate only the vectors in these selected in-verted lists. The search space is reduced by 2–3 orders of magnitude compared to a full corpus scan ( 𝑛𝑃𝑟𝑜𝑏𝑒 ≪ 𝑛𝐿𝑖𝑠𝑡 ). IVF is widely adopted in open-source ANN systems including FAISS [ 8] and ScaNN [ 15 ] due to its excellent scalability, low memory overhead, and flexible recall-performance trade-off. 

2.1.3 IVF Variants. Based on the storage format of vectors in inverted lists, three main IVF variants exist: 

IVF-Flat stores full-precision raw vectors directly in inverted lists. It achieves the highest possible recall but requires 𝑁 × 𝐷 × 4

bytes for FP32 vectors, limiting its applicability to small- to medium-scale datasets (up to ∼100 million vectors). 

IVF-PQ combines IVF indexing with PQ [ 20 ], compressing high-dimensional vectors into compact PQ codes (typically 8–32 bytes per vector) and storing the codes in inverted lists. It reduces memory footprint by 16–32 × compared to IVF-Flat, enabling billion-scale indices to fit in memory, but suffers from non-negligible accuracy loss due to quantization errors. 

IVF-RaBitQ. The IVF-RaBitQ algorithm adopted in this work re-tains the memory efficiency advantage of IVF-PQ while significantly reducing quantization accuracy loss to address the fundamental trade-off limitation of prior IVF variants. The RaBitQ quantization method and its integration into the IVF pipeline are detailed in Section 2.2. 

2.1.4 Alternative Index Structures. For completeness, we note two alternative index structures: 

Graph-based indexes (e.g., HNSW [ 32 ]) deliver superior recall-latency trade-offs for moderate-scale datasets, but suffer from 2– 3× higher memory overhead and poor scalability for billion-scale corpora. 

Disk-based indexes (e.g., DiskANN [ 19 ]) support trillion-scale datasets by offloading indices to SSD, but incur 10–100 × higher latency than in-memory IVF solutions.

## 2.3 Ascend NPU Architecture 

The Ascend NPU family adopts the Da Vinci heterogeneous archi-tecture, designed for high-throughput AI training and inference workloads [ 17 , 30 ]. Unlike CPUs and GPUs that employ hardware-managed cache hierarchies and uniform compute arrays, the Da Vinci architecture has three defining characteristics that directly shape our system design. An architectural overview is shown in Figure 1. 

2.3.1 Heterogeneous Compute Units at Two Levels. The NPU or-ganizes compute resources into two tiers of heterogeneity. At the device level, the throughput-oriented AI Core array is comple-mented by on-device AI CPUs—general-purpose scalar processors optimized for latency-sensitive, control-heavy tasks such as sorting and scheduling. Within each AI Core, two physically distinct execution units operate concurrently: the Cube Unit (AIC) for dense matrix multiplication and the Vector Unit (AIV) for element-wise operations. This two-level heterogeneity directly motivates our AIC-AIV fusion operator (Section 3.2.1), our computation flow restructuring (Section 3.2.2), and our decision to offload Top-

𝑘 selection to the AI CPU with intra-NPU pipeline parallelism (Section 3.3.3). 

2.3.2 Explicitly Managed On-Chip Buffers. Unlike GPUs where L1 data caches are hardware-managed, the Ascend NPU’s per-core Uni-fied Buffer (UB) and L1 Buffer are entirely software-managed. Every byte of data reaching the compute units must be explicitly loaded by the developer via Memory Transfer Engine (MTE) instructions; there is no automatic caching or prefetching at this level. Total on-chip buffer capacity per AI Core is approximately 1 MB. The L2 Cache, by contrast, is hardware-managed and transparent to the programmer, operating analogously to GPU L2 caches. This hybrid memory model is the primary reason existing CPU/GPU RaBitQ implementations cannot be directly ported—they rely on hardware-managed L1 caches to absorb irregular index access patterns—yet it also enables our index block-level load balancing and explicit on-chip staging of quantized vectors (Section 3.3.1), as well as our 1-bit vector computation adaptation (Section 3.3.2), which together eliminate HBM bandwidth from the inner loop. 

2.3.3 Dedicated Asynchronous Data Movement Engines. The MTE handles all data movement between HBM and on-chip buffers, operating fully asynchronously with respect to the AI Cores and AI CPUs. Combined with the Huawei Cache Coherence System (HCCS)—a shared-memory interconnect between NPU HBM and host CPU memory—this enables overlapped pipelines spanning both intra-NPU and NPU-CPU boundaries. We exploit this asyn-chrony to build the intra-NPU pipeline between AI Core and AI CPU (Section 3.3.3) and the cross-device pipeline between NPU and host CPU for fine re-ranking (Section 3.3.4). In summary, the Da Vinci architecture replaces the uniform compute and automatic caching paradigm of CPUs/GPUs with deliberate heterogeneity and explicit data orchestration. Section 3 describes how we translate each of these architectural properties into concrete system optimizations. 

## 3 Ascend-RaBitQ System Design 

Figure 2 illustrates the system architecture of Ascend-RaBitQ, which implements large-scale, efficient IVF-RaBitQ index construction and search based on a heterogeneous CPU-NPU collaborative system. Considering the billion-scale vector dataset size and limited HBM capacity, Ascend-RaBitQ adopts a 1-bit quantization design instead of multi-bit solutions similar to [ 37 ]. Overall, raw vector data is stored in the high-capacity CPU memory system to support billion-scale datasets. The CPU is responsible for global orchestration and re-ranking computations on raw vectors to minimize cross-chip data movement. The NPU’s HBM stores 1-bit quantized indexes and precomputed constants, and accelerates computation-intensive vector operations. Ascend-RaBitQ is fully optimized for the Ascend NPU hardware architecture, improving index construction and search efficiency through three key designs: fused AI Cube (AIC) and AI Vector (AIV) collaborative computing, multi-core load bal-ancing scheduling, and pipeline latency masking between AI Core and AI CPU. ## 3.1 Overall Design of Ascend-RaBitQ 

3.1.1 Index Construction. The index construction pipeline consists of three phases: the first two phases iteratively train IVF cluster centers using K-Means, and the third phase performs core RaBitQ quantization. Details are as follows: 

• Phase 1 : The CPU randomly generates nList initial cluster centers, and iteratively updates these centers based on the bucketing results returned from Phase 2. 

• Phase 2 : The vector dataset is partitioned into multiple batches according to HBM capacity, sequentially offloaded to the NPU for bucketing computation using the latest cluster centers, and results are transferred back to the CPU. Phases 1 and 2 iterate repeatedly until the K-Means termination condition (convergence or maximum iteration limit) is met. The core computation during bucketing is batch calculation of L2 distances between dataset vectors and cluster center vectors. We design an AIC-AIV fusion operator to improve computational efficiency, which is discussed in detail in Section 3.2.1. 

• Phase 3 : After bucketing information is finalized, RaBitQ quantization and constant precomputation are performed on all dataset vectors, including random orthogonal rota-tion, L1 distance calculation, and L2 norm calculation. To fully exploit the parallelism between AI Cube and AI Vector, we reorganize the traditional sequential computation flow, which is discussed in detail in Section 3.2.2. 

3.1.2 Search Pipeline. Ascend-RaBitQ processes queries at batch granularity. The first four phases perform coarse search on the NPU based on quantized indexes, and the final phase performs re-ranking on the CPU using raw vectors. Details are as follows: 

• Phase 1 : Batch computation obtains nProbe nearest cluster centers for each query vector. This process reuses the buck-eting computation flow from index construction Phase 2, fully utilizing both AIC and AIV computing power. 

• Phase 2 : After the CPU obtains the cluster centers for each query, it performs load balancing scheduling according to batch size, number of nProbe , and number of AIV cores to maximize AIV utilization, which is discussed in detail in Section 3.3.1. 

• Phase 3 : Distance calculation is performed, with the core computation being the dot product between query vectors and 1-bit quantized indexes. Based on the INT8 format, we adopt the high-order select operator capability provided by the CANN software stack to achieve efficient 1-bit computing, with details discussed in Section 3.3.2. 

• Phase 4 : Top-𝑘 selection, where we leverage AI CPU computing power for this scalar-dominant computation. Two optimizations are adopted: first, IVF-aware pruning is performed to accelerate sorting; second, pipeline par-allelism between AI Core and AI CPU (i.e., Phase 3 and Phase 4) is implemented within the batch to mask Top-𝑘 

latency, which is discussed in detail in Section 3.3.3. 

• Phase 5 : The CPU performs re-ranking based on raw vectors to select the final Top-𝑘 results. We discuss different re-ranking schemes and our optimized NPU-CPU pipeline latency masking implementation in Section 3.3.4. 

## 3.2 Index Construction Acceleration 

3.2.1 AIC and AIV Fusion Operator. When calculating L2 distances between a batch of dataset vectors (or query vectors) and a batch of cluster center vectors, we decompose the computation into two parallel parts: dot product between dataset vectors and cluster cen-ters, and norm calculation of each individual vector. The vector dot product is converted into matrix multiplication, where the left and right matrices are dataset vectors and cluster centers respectively, and we invoke CANN’s optimized matmul operator to run on AIC. The norm of each vector is calculated on AIV in parallel, and final L2 distance results are aggregated on AIV. We have developed a fused AIC-AIV operator that eliminates synchronization overhead between the two computing units, achieving parallel execution of dot product and norm calculation. 

3.2.2 Computation Flow Restructuring. In the standard RaBitQ pipeline, all vectors are first rotated by a random orthogonal matrix, then difference between dataset vectors and cluster center vectors (for sign extraction to obtain 1-bit quantized values), L1 distance, and L2 distance (precomputed constants) are calculated based on rotated vectors. Traditional implementations execute these computations sequentially since they share the same computing 

> 5

unit. Ascend-RaBitQ leverages the NPU’s heterogeneous computing capability to offload rotation computation to AIC, while decom-posing subsequent computations to run in parallel with rotation. Normally, subsequent computations require rotated vectors as input, but we exploit the key property of orthogonal matrices that rotation does not change vector norms. We therefore advance the norm calculation in L2 distance computation to run in parallel with rotation as the first stage computation. In the subsequent second stage, similar to Section 3.2.1, dot product computation and remaining computations are executed in parallel on AIC and AIV respectively. This restructured flow fully unleashes the parallel computing potential of both AIC and AIV. 

## 3.3 Retrieval Acceleration 

3.3.1 Load Balancing. Similar to industry IVF solutions such as FAISS [ 23 ], Ascend-RaBitQ divides each inverted list into segNum 

index blocks. Each index block is stored in contiguous physical addresses internally to improve memory access efficiency, while no contiguity requirement exists between blocks to improve memory utilization. When processing a batch of query vectors, a total of 

𝑏 ×𝑛𝑃𝑟𝑜𝑏𝑒 ×𝑠𝑒𝑔𝑁𝑢𝑚 index blocks need to be processed for distance calculation, where 𝑏 is the query batch size. Existing solutions gener-ally launch distance calculation kernels at query vector granularity: they allocate 𝑛𝑃𝑟𝑜𝑏𝑒 × 𝑠𝑒𝑔𝑁𝑢𝑚 computation tasks to coreNum AIV cores for parallel execution, and launch ⌈ 𝑛𝑃𝑟𝑜𝑏𝑒 ×𝑠𝑒𝑔𝑁𝑢𝑚 𝑐𝑜𝑟𝑒𝑁𝑢𝑚 ⌉ kernels sequentially for each query. When 𝑛𝑃𝑟𝑜𝑏𝑒 × 𝑠𝑒𝑔𝑁𝑢𝑚 is not an integer multiple of coreNum , AIV cores will be idle during the last kernel launch for each query, wasting computing power, as illustrated in Figure 3 (a). Ascend-RaBitQ breaks the query vector boundary and performs load balancing scheduling at index block granularity. Each kernel launch schedules coreNum index blocks and evenly allocates them to AIV cores for computation, as shown in Figure 3 (b). In this design, AIV computing power waste may only occur during the very last kernel launch for the entire batch, which greatly improves NPU utilization, reduces the total number of kernel launches, and further shortens end-to-end latency. We considered an alternative optimization that schedules index blocks from different queries that access the same inverted list into the same kernel launch, reducing index block transfer between the unified buffer and HBM. However, this approach completely disrupts query order, making it impossible to implement pipeline parallelism with subsequent modules as designed in Section 3.3.3 and 3.3.4, ultimately leading to reduced end-to-end performance. Therefore, Ascend-RaBitQ maintains query order when scheduling index blocks. 

3.3.2 1-bit Vector Computation Adaptation. There are two main categories of distance calculation schemes for RaBitQ: 1) utilizing hardware native 1-bit operation capabilities (such as the POPCNT instruction on CPUs) to directly calculate the dot product between query vectors and 1-bit index vectors (CPU implementations require additional scalar quantization of query vectors into multiple 1-bit vectors); 2) the FastScan scheme, which precomputes Look-Up Tables (LUTs) and obtains dot product results through table lookup [ 2, 3 ]. 3.3.3 Top-𝑘 Optimization. The Ascend NPU integrates a dedicated AI CPU, which is more suitable for processing scalar-dominant computations compared to the AI Core optimized for parallel vector operations. To fully utilize all NPU computing resources, Ascend-RaBitQ offloads Top-𝑘 selection to AI CPU and introduces two targeted optimization techniques. First, when traversing all distance calculation results for sorting, we maintain an upper bound in real time, which is the maximum distance in the current Top-𝑘 candidate set. Remaining distances larger than this upper bound are pruned and excluded from further sorting. We exploit the key IVF property that vectors in inverted lists of cluster centers closer to the query vector are significantly more likely to enter the final Top-𝑘 set. Therefore, we traverse inverted lists in the order of the distance between the query vector and their corresponding cluster centers, prioritizing lists of closer clusters to maximize pruning efficiency. 

> 6

The second optimization is inter-unit pipeline parallelism be-tween AI Core and AI CPU, as illustrated in Figure 5 (a) and (b). Specifically, within a batch, when a query vector completes distance calculation on AI Core, a dedicated flag in HBM is atomically set. The AI CPU continuously scans these flags and immediately starts Top-𝑘 computation for a query as soon as its corresponding flag is set. Our evaluation shows that this pipeline parallelism effectively masks nearly all Top-𝑘 latency, shortening end-to-end retrieval latency significantly. 

3.3.4 NPU-CPU Pipeline Parallelism. Ascend-RaBitQ adopts ex-treme 1-bit quantization to fully utilize limited HBM capacity, re-quiring a final re-ranking step on raw vectors to achieve acceptable recall rate. One candidate re-ranking approach is to send required raw vectors from CPU to NPU, and perform re-ranking on NPU to leverage its high vector computing throughput. The alternative is to perform re-ranking on CPU directly, avoiding cross-device data transfer. ## 3.4 Distributed Scaling 

As vector dataset sizes continue to grow or throughput require-ments increase further, single NPU devices will eventually en-counter bottlenecks in memory capacity or computing resources. To address this challenge, Ascend-RaBitQ is designed with na-tive distributed scaling capability. We adopt a cluster-aware load balancing strategy to evenly partition quantized vector datasets across multiple NPU devices, ensuring that the number of vectors in each inverted list is balanced across all nodes. During the index construction phase, each NPU independently performs quantization operations on its responsible vector shard in parallel, achieving near-linear speedup with increasing number of devices. During 

> 7

retrieval, multiple NPU devices simultaneously process the same query batch, independently performing their local coarse search computation tasks. Finally, the CPU aggregates results from all NPU nodes and performs global re-ranking to output the final accurate Top-𝑘 retrieval results. 

## 4 Evaluation 

We conduct extensive experiments to evaluate the performance of our proposed Ascend-RaBitQ system on a variety of standard vector search datasets, comparing against state-of-the-art CPU IVF quantization implementations and the native NPU full-precision baseline. All experiments are performed on a server equipped with 8 Ascend NPUs and a host CPU. Each NPU features 48 AIV cores and 24 AIC cores clocked at 1.85 GHz, with 64 GB HBM. The host CPU has 192 cores at 2.6 GHz and 1.5 TB DDR4 memory. Unless otherwise specified, experiments use a single NPU together with the host CPU. 

## 4.1 Experimental Setup 

Datasets. We use four widely adopted public datasets covering different application domains (image, text) and data scales ranging from 1 million to 1 billion vectors: 

• SIFT1M [ 20 ]: 1 million 128-dimensional SIFT descriptors for image retrieval (small-scale image workload); 

• Cohere 10M [ 18 ]: 10 million 768-dimensional text embed-dings (large-scale text semantic search workload); 

• SIFT100M [ 20 ]: 100 million 128-dimensional SIFT descrip-tors (medium-scale hundred-million-level candidate image workload); 

• SIFT1B [ 23 ]: 1 billion 128-dimensional SIFT descriptors (large-scale billion-level production workload). Baselines. We compare against two categories of state-of-the-art baselines for fair evaluation: (1) Same NPU platform baseline: Ascend IndexSDK IVF-Flat implementation, the native full-precision vector search solution on Ascend NPU; (2) CPU platform baselines: Three standard IVF-RaBitQ im-plementations from FAISS library (v1.14.1) [ 8 ] with dif-ferent quantization configurations, each optimized with architecture-specific instructions for the target CPU plat-form: 

• CPU 1-bit : Adopts 1-bit RaBitQ quantization for database vectors only, keeping query vectors in full precision. This variant is mathematically equivalent 

to our Ascend-RaBitQ, making it the fairest baseline for evaluating the raw compute advantage of NPU acceleration over CPU. 

• CPU 1-bit+SQ : Applies 4-bit scalar quantization (SQ) to query vectors while keeping database vectors at 1-bit RaBitQ quantization. The query-side SQ enables the CPU to efficiently compute inner products between two low-precision vectors using SIMD instructions, significantly accelerating distance computation. 

• CPU 4-bit : Quantizes database vectors to 4-bit using RaBitQ. This configuration achieves higher precision than the 1-bit variants at the cost of increased memory access and computation, serving as a reference for balanced quantization-performance tradeoffs. All methods adopt identical IVF configurations for fair com-parison, with 𝑛𝐿𝑖𝑠𝑡 = √𝑁 and 𝑛𝑃𝑟𝑜𝑏𝑒 = 5% · 𝑛𝐿𝑖𝑠𝑡 , where 𝑁

## 4.2 NPU Platform Native Comparison: Ascend-RaBitQ vs. IVF-Flat 

We conduct a comprehensive performance comparison between Ascend-RaBitQ and the native Ascend IndexSDK IVF-Flat imple-mentation on the Ascend NPU platform. An important practical consideration motivates our experimental design here: the full-precision IVF-Flat index requires storing the entire dataset in NPU memory, which imposes significant memory constraints. For a single Ascend NPU with 64 GB HBM, only the SIFT1M dataset (512 MB full-precision index) can be accommodated. Larger datasets such as SIFT100M (51 GB) and SIFT1B (512 GB) exceed single-NPU memory capacity, making single-NPU IVF-Flat comparisons infeasible. Therefore, we first compare Ascend-RaBitQ against IVF-Flat on a single NPU using the SIFT1M dataset, then extend the comparison to four NPUs using the SIFT100M dataset. Figure 6 presents the single-NPU (1 × Ascend) performance comparison on the SIFT1M dataset. Ascend-RaBitQ demonstrates consistent performance improvements across all three Top-𝑘 set-tings at the same recall. Ascend-RaBitQ achieves a 17.6 × QPS speedup over IVF-Flat at recall = 0.95 under both top-10 and top-100 settings, and 12.7 × under top-300. This performance advan-tage originates from two key architectural factors. First, our 1-bit quantization reduces memory access volume by 32 × compared to full-precision IVF-Flat, effectively eliminating the HBM bandwidth bottleneck even when retrieving significantly more candidates for higher Top-𝑘 targets. Second, while both methods leverage the Cube unit for matrix multiplication-based distance computation, Ascend-RaBitQ’s 1-bit quantized vectors enable substantially lower computational complexity than IVF-Flat’s full-precision vector distance calculation. For real-time query latency on a single NPU, Ascend-RaBitQ achieves an average latency of 0.34 ms with a small batch size of 64, representing a order-of-magnitude reduction compared to IVF-Flat’s 5.98 ms latency. This enables the system to meet strict low-latency online serving requirements. In terms of storage and memory requirements, Ascend-RaBitQ achieves a 16 × theoretical compression ratio compared to full-precision IVF-Flat, with actual storage savings of 93.75%. For the SIFT1M dataset, the IVF-Flat index occupies 512 MB of disk storage and requires all data to be resident in memory during query processing, while the Ascend-RaBitQ index only requires 32 MB including IVF cluster metadata and quantization codebook. ## 4.4 Ablation Study 

We validate the effectiveness of each retrieval acceleration tech-nique (detailed in Section 3.3) through controlled ablation experi-ments on the SIFT1B dataset, as shown in Figure 9. The baseline adopts the naive IVF-RaBitQ implementation with the CANN 

select operator for distance computation, without inter-core load balancing, without pipeline parallelism between distance calcula-tion and Top-𝑘 selection, and with re-ranking performed on the NPU. We measure the performance impact by incrementally adding each optimization on top of this baseline. First, adding our index block-level cross-query load balancing scheduling (Section 3.3.1) achieves a 6.3 × speedup over baseline. The optimized scheduling eliminates idle AIV cores caused by uneven task allocation in the original query-level kernel launch mechanism. Second, replacing the select operator with the gather-based FastScan scheme for 1-bit vector distance calculation (Section 3.3.2) further achieves a 15.2 × cumulative speedup. The FastScan approach achieves higher computational efficiency by leveraging precomputed look-up tables for 1-bit vector distance calculation. Third, adding AI CPU offloaded Top-𝑘 pruning and intra-NPU AI Core-AI CPU pipeline parallelism (Section 3.3.3) further achieves a 16.5 × cumulative speedup. The pipeline design effectively masks nearly all Top-𝑘 selection latency, which previously accounted for a significant portion of total end-to-end runtime. Finally, moving re-ranking from NPU to host CPU (Section 3.3.4) further achieves a 23.1 × cumulative speedup over baseline. This eliminates the CPU data gathering bottleneck from discontinuous memory, as the re-ranking vectors are already resident in CPU memory. All four retrieval acceleration techniques combined contribute to the overall 23.1 × speedup over the ablation baseline, demon-strating that each optimization proposed in Section 3.3 is critical to achieving the final performance. 

## 4.5 Multi-NPU Large-scale Scalability Test 

Finally, we evaluate the scalability of Ascend-RaBitQ on the billion-scale SIFT1B dataset using up to 8 Ascend NPUs to demonstrate its suitability for large-scale industrial deployment. Figure 10 presents the speedup of end-to-end throughput and raw distance computation throughput as the number of NPUs increases. As shown in the figure, the end-to-end throughput achieves only approximately 3 × speedup on 8 NPUs compared to a single NPU, falling short of linear scaling. This is because NPU acceleration primarily benefits the coarse ranking stage, while the overall throughput remains constrained by CPU re-ranking. As more NPUs are added, the system bottleneck progressively shifts from NPU computation to CPU post-processing. This finding suggests that introducing CPU pipeline parallelism to overlap re-ranking with NPU computation could further improve end-to-end throughput. In contrast, the distance computation stage scales more linearly, achieving approximately 4 × speedup on 8 NPUs. However, this still reaches only about half of the theoretical optimal value of 8 ×.Our in-depth analysis of the experimental data reveals two pri-mary causes for the sub-linear scaling of distance computation. First, we manage vectors at the granularity of index blocks, where the total number of blocks determines the number of kernel launches. Even when inverted lists are distributed across different NPUs, the total number of blocks may not decrease linearly, leading to sub-optimal kernel launch efficiency at scale. Second, the vector counts across inverted lists are inherently uneven due to the data distribution. Although we employ block-level load balancing, when blocks from different lists are computed concurrently, inter-core load imbalance can still occur, further reducing linearity. Two potential directions can address these issues. First, introduc-ing list-level load balancing during clustering training could ensure a more balanced vector distribution across lists. Second, using a smaller block size would improve the granularity of block-level load balancing. However, smaller blocks lead to more fragmented mem-ory storage, degrading memory access efficiency, and the increased number of blocks adds management complexity. Determining the optimal block size for different workload scenarios is an interesting research problem and remains as our future work.

## 6 Conclusion 

This paper presents Ascend-RaBitQ, the first NPU-accelerated IVF-RaBitQ system designed for heterogeneous NPU-CPU architectures. By decomposing vector search into a three-stage collaborative pipeline—AI Core coarse ranking, AI CPU Top-𝑘 selection, and host CPU fine re-ranking—our approach maps each stage to its optimal hardware, resolving the accuracy-memory-performance trilemma. Extensive experiments demonstrate that Ascend-RaBitQ achieves 3.0 × to 62.8 × faster index construction, up to 4.6 × through-put improvement over the fastest CPU implementation, and over 100 × over the mathematically equivalent CPU baseline. Controlled ablation studies confirm a cumulative 23.1 × speedup, and multi-NPU tests demonstrate scalable throughput on distributed configu-rations. We are actively extending Ascend-RaBitQ to trillion-scale cross-NPU-node deployments, deepening integration with production vector databases such as Milvus [ 42 ] and Faiss [ 8], and optimizing for low-latency single-query scenarios.
