## 摘要
在边缘设备上本地执行 AI 对于低延迟和离线运行至关重要。然而，在多样化的硬件上部署模型仍然存在碎片化问题，通常需要在最初编写模型的 PyTorch 生态系统之外进行模型转换或完全重新实现。我们介绍了 ExecuTorch，这是一个用于边缘 AI 的统一 PyTorch 原生部署框架。ExecuTorch 能够在异构计算环境中无缝部署机器学习模型。它的扩展范围涵盖了从嵌入式微控制器到带有专用加速器的复杂系统级芯片（SoC），为从可穿戴设备、智能手机到大型计算集群的各种设备提供动力。ExecuTorch 在保留 PyTorch 语义的同时，允许进行定制，支持量化等优化，并提供可插拔的执行“后端”。这些特性共同实现了快速实验，使研究人员能够完全在 PyTorch 内部验证部署行为，从而弥合了研究与生产之间的差距。

## Abstract
Local execution of AI on edge devices is important for low latency and offline operation. However, deploying models on diverse hardware remains fragmented, often requiring model conversion or complete reimplementation outside the PyTorch ecosystem where the model was originally authored. We introduce ExecuTorch, a unified PyTorch-native deployment framework for edge AI. ExecuTorch enables seamless deployment of machine learning models across heterogeneous compute environments. It scales from embedded microcontrollers to complex system-on-chips (SoCs) with dedicated accelerators, powering devices ranging from wearables and smartphones to large compute clusters. ExecuTorch preserves PyTorch semantics while allowing customization, support for optimizations like quantization, and pluggable execution "backends". These features together enable fast experimentation, allowing researchers to validate deployment behavior entirely within PyTorch, bridging the gap between research and production.

### 2. 方法论：核心思想与关键技术
ExecuTorch 的架构分为 **AOT（提前编译）导出栈** 和 **Runtime（运行时）栈** 两部分：
*   **核心思想**：通过提前将动态的 PyTorch 模型捕获为静态图，并针对特定硬件进行优化，从而在保持 PyTorch 语义的同时，降低运行时的开销。
*   **关键技术细节**：
    *   **torch.export & Edge Dialect**：利用 PyTorch 2.0 的导出技术，将模型转换为仅包含不到 300 个核心算子（Core ATen）的“边缘方言”图，去除了对 Python 的依赖。
    *   **后端委派（Backend Delegation）**：允许将图的子部分委派给专用加速器（如 NPU、GPU、DSP）。目前支持 XNNPACK (CPU)、Vulkan (GPU)、Qualcomm QNN、Apple CoreML、Arm Ethos-U 等。
    *   **内存规划（Memory Planning）**：在 AOT 阶段计算张量的生命周期和大小，预先分配静态内存池，避免运行时的动态内存分配。
    *   **量化支持**：集成 TorchAO，支持从 8-bit 静态量化到 LLM 专用的 4-bit 组量化（Group-wise quantization）。
    *   **序列化格式**：引入 `.pte`（程序执行）和 `.ptd`（数据/权重）文件格式，支持权重共享和多方法执行。

### 6. 主要结论与发现
*   **性能领先**：在 CPU 上通过 XNNPACK 表现强劲；在移动端 GPU 上通过 Vulkan 实现了极高的 Token 生成吞吐量；在 NPU 上通过 QNN 显著提升了 LLM 的 Prefill 速度。
*   **实验一致性**：研究人员可以在 PyTorch 内部验证量化和算子行为，确保在手机上运行的结果与服务器端模拟几乎完全一致。
*   **生产可用性**：ExecuTorch 已在 Meta 旗下的应用（如 Ray-Ban 智能眼镜、手机端 App）中支撑了每日数十亿次的推理请求。
*   **极低门槛**：成功在不到 1 美元的微控制器（Pico 2）上运行了量化模型，内存占用仅约 11 KiB。

### 7. 优点
*   **原生集成**：深度集成于 PyTorch，无需复杂的模型转换工具链。
*   **高度可扩展**：后端委派机制非常灵活，硬件厂商可以轻松接入自己的 SDK。
*   **极致轻量**：核心运行时采用 C++17 编写，无动态分配，非常适合资源受限的嵌入式环境。
*   **全栈优化**：从 AOT 内存规划到运行时的算子融合，提供了端到端的性能保障。

### 8. 不足与局限
*   **导出限制**：依赖 `torch.export`，对于包含复杂动态控制流（如数据依赖的循环）或动态形状（Dynamic Shapes）的模型，导出过程仍具挑战，可能需要重写部分模型代码。
*   **硬件适配成本**：虽然支持 AOT 优化，但针对不同厂商的 NPU 仍需生成特定的 `.pte` 文件，增加了多设备分发的工程复杂度。
*   **桌面端支持尚在早期**：目前主要侧重于移动端和嵌入式，对桌面端 GPU（CUDA/Metal）的支持仍处于实验阶段。
*   **稀疏化支持不足**：虽然 IR 支持稀疏表示，但目前大多数边缘硬件后端尚未广泛实现硬件加速的稀疏算子。
