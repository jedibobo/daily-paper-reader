## 摘要
大语言模型（LLM）服务目前受限于键值（KV）缓存。在解码过程中，每个新生成的 token 都会重新读取之前的 KV 状态，因此注意力机制（attention）变成了一项对带宽和容量要求极高的内存任务。HBM-PIM 通过将注意力计算移至靠近内存的位置来提供帮助，但目前的堆栈组织方式仍然存在资源浪费。实际上，只有热点 KV 块能从近存计算中获益。权重、激活值和冷 KV 块主要需要高密度存储和 GPU 可见的带宽。统一的 HBM-PIM 堆栈使得所有层都要为 PIM 逻辑付出代价，而像 AttAcc 这样的专用 PIM 设计虽然恢复了容量，但却缩减了留给 GPU 端工作的 HBM 带宽。我们提出了 TokenStack，这是一种针对以 KV 为中心的 LLM 服务而设计的垂直异构 HBM-PIM 架构，它利用了 HBM4 的逻辑裸片（logic-die）基底。TokenStack 将每个堆栈分为高密度容量层和支持 PIM 的计算层，然后将逻辑基底裸片作为堆栈本地控制点，在没有主机端开销的情况下管理跨层数据移动。基底裸片控制器负责处理跨层 DMA、分层地址转换、注意力端的聚合/广播协调以及迁移过程中的内联量化。在该硬件之上，TokenStack 采用拓扑感知的 KV 放置、工作负载感知的驱逐和有界复制策略，以将热点 KV 保持在 PIM 计算附近，同时将较冷的 KV 状态移动到高密度层。使用来自四个模型的生产环境轨迹进行的多 QPS 运行结果显示，与 AttAcc 相比，TokenStack 将几何平均 token 吞吐量提高了 1.62 倍，将符合 SLO 的服务容量提高了 1.70 倍，并将每个 token 的能耗降低了 30-47%。

## Abstract
Large language model (LLM) serving is now limited by the key-value (KV) cache. During decode, each new token rereads prior KV state, so attention becomes a bandwidth- and capacity-heavy memory task. HBM-PIM helps by moving attention closer to memory, but current stack organizations still waste resources. In practice, only hot KV blocks benefit from near-memory compute. Weights, activations, and cold KV mainly need dense storage and GPU-visible bandwidth. A uniform HBM-PIM stack makes all layers pay for PIM logic, while a dedicated-PIM design such as AttAcc recovers capacity but shrinks the HBM bandwidth left for GPU-side work. We propose TokenStack, a vertically heterogeneous HBM-PIM architecture for KV-centric LLM serving that leverages HBM4's logic-die substrate. TokenStack separates each stack into dense capacity layers and PIM-enabled compute layers, then uses the logic base die as a stack-local control point that manages cross-layer movement without host-side overhead. The base-die controller handles cross-layer DMA, layered address translation, attention-side gather/broadcast coordination, and inline quantization during migration. On top of this hardware, TokenStack uses topology-aware KV placement, workload-aware eviction, and bounded replication to keep hot KV near PIM compute while moving colder state to dense layers. Using production-derived traces across four models, completed multi-QPS runs show that TokenStack increases geometric-mean token throughput by 1.62x and SLO-compliant serving capacity by 1.70x over AttAcc, and reduces per-token energy by 30-47%.

### 1. 核心问题与研究动机
*   **核心问题**：大语言模型（LLM）推理的解码阶段（Decode）受限于键值（KV）缓存带来的**带宽**和**容量**双重瓶颈。
*   **研究背景**：
    *   解码阶段的算术强度极低（$O(1)$），导致注意力机制计算完全受限于内存带宽。
    *   现有的近存计算（PIM）方案存在缺陷：**统一架构（Uniform）**让所有存储层都携带PIM逻辑，导致存储密度减半，浪费了存储权重和冷数据的空间；**专用架构（Dedicated，如AttAcc）**虽然保留了部分高密度层，但减少了GPU可见的HBM总带宽，拖慢了预填充（Prefill）和非注意力层的执行。
*   **动机**：KV缓存具有显著的异构性（少量热块贡献大部分重用），因此需要一种物理上异构的存储堆栈来匹配这种数据特性。

### 2. 方法论：TokenStack 架构与运行时
TokenStack 提出了软硬件协同的异构设计：
*   **垂直异构堆栈（Hardware）**：
    *   将HBM堆栈垂直划分为：**容量层（Capacity Layers）**，采用标准高密度DRAM，存储权重、激活值和冷KV；**计算层（Compute Layers）**，集成PIM引擎，存储热KV块。
    *   **逻辑底座（Logic Base Die）**：利用HBM4标准，在底座集成控制器，负责跨层DMA、分层地址转换、注意力协调以及**内联K8V4量化**（在数据从计算层移向容量层时自动压缩）。
*   **数据布局策略**：
    *   **Key缓存**：采用Token-major布局，便于PIM并行计算评分。
    *   **Value缓存**：采用Dim-head布局，便于并行计算输出。
*   **运行时优化（Runtime）**：
    *   **拓扑感知放置**：将请求分配到其前缀（Prefix）所在的物理卡/堆栈。
    *   **类别感知驱逐**：根据工作负载类型（如API、推理、代码）预测KV块的重用概率，将低价值块通过堆栈内DMA“降级”到容量层。
    *   **有界复制**：仅对高频访问的系统提示词（System Prompts）进行跨卡复制，平衡通信开销与访问收益。

### 3. 实验设计
*   **数据集/场景**：使用了4种源自阿里云生产环境（Qwen-Bailian）的真实轨迹（Trace），涵盖API/文本、混合、代码、长文本推理四种模式。
*   **模型**：Qwen3-4B、Qwen3-32B、Mistral-Devstral-123B、GPT-175B。
*   **Benchmark与对比方法**：
    *   **Full-GPU**：无PIM的纯GPU基准。
    *   **Uniform**：所有层均支持PIM的统一架构。

*   **评估指标**：Token吞吐量、SLO达标率、端到端延迟（TTFT/TBT）、每Token能耗。

### 4. 资源与算力
*   **硬件环境**：模拟了DGX-A100平台，包含8个GPU，配备HBM3/HBM4级别的存储规格。
*   **算力细节**：GPU峰值算力设定为312 TF/s，NVLink3带宽600 GB/s，UCIe-A带宽512 GB/s。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了16组“模型-轨迹”组合，并进行了QPS（每秒请求数）负载扫描。
*   **消融实验**：详细分析了数据布局、调度策略、量化引擎、驱逐算法等各个组件对性能的贡献。
*   **充分性评价**：实验设计非常全面，不仅对比了吞吐量，还深入分析了排队延迟分布和能量消耗组成。通过不同模型规模的对比，验证了方案在KV压力增大时的扩展性，结果具有较强的说服力和客观性。

### 6. 主要结论与发现
*   **性能提升**：相比AttAcc，TokenStack 的几何平均Token吞吐量提升了**1.62倍**，在123B/175B等大模型上提升更显著（最高2.32倍）。
*   **容量优势**：在满足2倍延迟SLO的前提下，服务容量提升了**1.70倍**。
*   **能效优化**：通过减少昂贵的离片（Off-chip）内存访问，每Token能耗降低了**30-47%**。
*   **关键发现**：异构堆栈能有效缓解“容量-带宽”矛盾；内联量化和拓扑感知调度是维持高PIM命中率的关键。

### 7. 优点与亮点
*   **前瞻性架构**：紧扣HBM4技术趋势，巧妙利用逻辑底座（Base Die）解决PIM管理开销问题。
*   **软硬协同**：不只是硬件堆叠，还针对LLM数据流设计了非对称的KV布局和基于生产环境特征的驱逐算法。
*   **资源利用率高**：通过垂直异构设计，既保留了GPU所需的全局带宽，又最大化了KV缓存的有效容量。

### 8. 不足与局限
*   **应用限制**：在KV重用率极低（如纯长文本推理且无前缀共享）或模型极小（KV不溢出）的场景下，该架构带来的增益会显著收缩（如4B模型仅提升3%）。
*   **硬件依赖**：高度依赖HBM4及UCIe等先进封装技术，短期内商业化落地成本较高。
*   **模拟偏差**：虽然使用了周期精确模拟，但实际硬件中的热管理（Thermal）和制造良率（Yield）对异构堆栈的影响在文中仅做了初步讨论，可能存在工程实现上的挑战。
