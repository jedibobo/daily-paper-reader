## 摘要
随着边缘计算的扩展，在单个共享 GPU 上提供多个深度神经网络 (DNN) 模型服务已成为一种常见但具有挑战性的场景，其中每个调度决策都会影响所有并发队列的尾部延迟。现有的调度器依赖于局部启发式方法，无法捕捉这种全局影响，而 GPU 空间共享方法则牺牲了延迟的可预测性。在本文中，我们提出了 EdgeServing，这是一种针对边缘设备的截止日期感知多 DNN 服务系统。EdgeServing 采用带有早退推理的时分 GPU 共享以实现高推理可预测性，并引入了稳定性评分来量化每个候选调度决策对未来队列状态的影响。在运行时，它协同选择模型、退出点和批处理大小，以最小化预测的全系统 SLO 影响。在多个硬件平台上的实验结果表明，EdgeServing 在 SLO 违规率和 P95 延迟方面均始终优于代表性基准方案，这得益于早退机制在严格延迟约束下扩展了调度动作空间。

## Abstract
As edge computing expands, serving multiple deep neural network (DNN) models on a single shared GPU has become a common yet challenging scenario, where each scheduling decision affects the tail latency of all concurrent queues. Existing schedulers rely on local heuristics and fail to capture this global impact, while GPU spatial-sharing approaches sacrifice latency predictability. In this paper, we propose EdgeServing, a deadline-aware multi-DNN serving system for edge devices. EdgeServing adopts time-division GPU sharing with early-exit inference for high inference predictability, and introduces a stability score to quantify how each candidate scheduling decision impacts the future queue status. At runtime, it cohesively selects the model, exit point, and batch size to minimize predicted system-wide SLO impact. Experimental results on multiple hardware platforms show that EdgeServing consistently outperforms representative baselines in both SLO violation ratio and P95 latency, enabled by early-exit mechanism, which expands the scheduling action space under tight latency constraints.

### 1. 核心问题与研究动机
*   **核心问题**：在边缘设备（如自动驾驶、AR设备）上，通常只有一个 GPU 需要同时处理多个 DNN 任务（如目标检测、分割等）。
*   **研究动机**：
    *   **现有共享机制的缺陷**：GPU 空间共享（如 NVIDIA MPS）会导致缓存和带宽竞争，使推理延迟变得不可预测；而传统的时间片轮转或局部启发式调度（如最长队列优先）忽略了当前决策对其他模型队列未来状态的全局影响。
    *   **延迟与精度的矛盾**：在流量激增时，全量模型推理往往无法在截止日期（Deadline）前完成，导致 SLO 违规。
    *   **早退机制的潜力**：虽然早退（Early-exit）技术能降低延迟，但现有研究多针对单模型，缺乏在多模型并发场景下协同优化“模型选择、退出点、批处理大小”的全局调度方案。

### 2. 方法论
EdgeServing 采用**时分 GPU 共享**结合**动态早退机制**，核心思想是“预测未来，最小化全局损害”。
*   **离线分析（Offline Profiling）**：在部署前，对所有模型在不同退出点、不同批处理大小（Batch Size）下的延迟（P95）和精度进行基准测试，生成查询表。
*   **稳定性评分（Stability Score）**：引入一个非线性函数来量化任务的紧迫性：$f(w) = \min(\exp(w/\tau - 1), C)$。其中 $w$ 是排队时间，$\tau$ 是 SLO 阈值。该评分随时间接近截止日期呈指数级增长，迫使调度器优先处理即将违规的任务。
*   **在线调度算法**：
    1.  **批处理与退出点选择**：针对每个队列，根据当前排队最久的任务，从离线表中查找能在截止日期前完成的最深（精度最高）退出点和最大批处理大小。
    2.  **状态预测**：模拟如果执行某个模型的决策，其他所有队列在推理完成后的新状态（排队时间增加）。
    3.  **全局决策**：计算所有决策后的全局稳定性评分，选择使系统整体最“稳定”（评分最低）的模型进行推理。

*   **流量模型**：采用泊松分布模拟请求到达，设置不同的流量强度（20 到 240 req/s）。

### 6. 主要结论与发现
*   **SLO 保证能力强**：在重负载下，EdgeServing 的 SLO 违规率保持在 1% 以下，而 All-Final 基准在相同条件下违规率超过 15%。
*   **精度与延迟的平滑权衡**：系统能根据流量自动调节退出深度。流量低时使用全量模型保证精度（约 77%），流量极高时退而求其次使用浅层退出点，虽精度下降但保证了系统不崩溃。
*   **全局优化的必要性**：消融实验证明，仅靠“最早截止日期优先（EDF）”或“最长队列优先（LQF）”无法达到 EdgeServing 的稳定性，证明了预测未来队列状态的价值。

### 7. 优点与亮点
*   **确定性延迟**：通过时分共享和离线 Profile，规避了 GPU 内部资源竞争带来的延迟抖动，这对实时系统至关重要。
*   **调度空间扩展**：将“早退点”引入调度决策维度，使得系统在严格延迟约束下拥有了更大的操作余地。
*   **通用性**：系统设计不依赖于特定模型架构，且能快速迁移到不同的边缘硬件平台。

### 8. 不足与局限
*   **离线依赖**：系统高度依赖离线 Profile 的准确性。如果运行时环境发生变化（如 GPU 降频、其他后台进程竞争），预设的延迟表可能失效。
*   **模型类型限制**：实验仅基于 CNN（ResNet），未探讨在 Transformer 等其他架构或多模态模型上的适用性。
*   **精度损失**：虽然保证了延迟，但在极端高负载下，系统会频繁选择 layer1 退出，此时精度（约 7%-9%）可能导致业务逻辑失效（如自动驾驶无法识别物体）。
*   **调度开销**：虽然调度器在 CPU 上运行，但随着模型数量和退出点增加，搜索最优决策的计算开销可能会成为新的瓶颈。
