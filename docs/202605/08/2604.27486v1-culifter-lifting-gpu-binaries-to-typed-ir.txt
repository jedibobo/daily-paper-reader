## Abstract 

GPU compilers merge all data types into a single unified register file, erasing the type information that binary-analysis tools rely on. We show that type recovery from this untyped register file is the central challenge of GPU binary lifting. We present CuLifter, a SASS-to-LLVM IR lifting framework that recovers register types via constraint propagation with conflict detection, reconstructs explicit control flow, and aggregates multi-instruction patterns. Across eight benchmark suites (24,437 GPU functions in 919 cubins) spanning open-source applications, vendor libraries, and optimized ML runtimes, CuLifter successfully lifts 99.98% of functions to valid LLVM IR. An ablation study confirms that type recovery is the only step required to produce semantically correct IR: disabling it drops the x86 pass rate from 73.8% to 0%, a 73.8 percentage-point drop. 

## Keywords 

GPU binary lifting, LLVM IR, SASS, type recovery, program analysis 

## 1 Introduction 

NVIDIA GPUs dominate high-performance and parallel computing, and CUDA [ 44 ] has become the de facto standard for GPU pro-gramming across ML frameworks (e.g., PyTorch [ 48 ], vLLM [ 35 ]) and HPC components [ 40 ]. Performance-critical CUDA kernels in-creasingly ship as pre-compiled SASS binaries without source code, including vendor libraries (cuDNN, TensorRT, cuBLAS, NCCL), optimized ML runtimes, and closed-source HPC components. Binary lifting translates machine code back into a higher-level IR amenable to compiler analysis. For CPU binaries, lifted IR enables a wide range of applications: security analysts use it to detect vulner-abilities in closed-source software [ 9 ], malware researchers apply taint analysis and symbolic execution on lifted code [ 7 ], and sys-tems researchers retarget legacy binaries to new architectures [ 13 ]or harden them with control-flow integrity [ 24 ]. These applications are supported by a mature tool ecosystem: static lifters such as McSema [ 19 ] and rev.ng [ 17 ] recover LLVM IR from x86, dynamic lifters such as BinRec [ 4] capture execution traces for recompilation, and a recent systematization of knowledge evaluates these tools across downstream applications [37]. No comparable infrastructure exists for GPU binaries. NVIDIA provides disassemblers ( nvdisasm , cuObjDump ), but these output only flat, untyped assembly without control-flow or type infor-mation. NVBit [ 57 ] and SASSI [ 53 ] instrument SASS dynamically and at compile time, respectively, but neither lifts the binary to an analyzable IR. CuAsmRL [ 26 ] optimizes SASS instruction sched-ules via reinforcement learning, and CASS [ 28 ] transpiles SASS to AMD RDNA3 assembly using LLMs, achieving 37.5% assembly-level translation accuracy. Both operate on SASS but neither produces a typed, retargetable IR suitable for compiler analysis. As a result, the analyses routinely applied to CPU binaries (vulnerability detection,

A key obstacle to GPU binary lifting is type recovery : determin-ing the data type (e.g., Float32 , Int32 , pointer) that each register holds at each program point. Type recovery is a well-known chal-lenge for CPU binaries as well, where integers and pointers share the same general-purpose registers and memory aliasing obscures data types. A long line of work addresses this problem, progressing from constraint-based inference [ 20 , 36 ] to lattice-based propa-gation [ 41 ], probabilistic prediction [ 61 ], and behavior-capturing types [ 6 ]. However, CPU ISAs still provide partial structural cues: separate register files for floating-point and vector values, explicit transfer instructions ( movd , cvtsi2sd ), and calling conventions that constrain parameter types. GPU ISAs provide far fewer cues. SASS stores all values in a single unified register file, and the same register can be consumed as Float32 by one instruction and as 

Int32 by the next, with no intervening transfer. Without recov-ering these implicit type boundaries, a lifter cannot emit IR at all: every LLVM IR instruction requires explicit type annotations, so unresolved types block code generation entirely. SASS [ 42 ] is the native machine code for NVIDIA GPUs. During compilation, CUDA source code is first compiled to PTX [ 45 ], a hardware-independent virtual assembly that preserves structured control flow and type information, and then lowered to SASS, a hardware-specific assembly where much of this structure is lost. SASS is optimized for specific GPU architectures (e.g., Turing SM75, Ampere SM80, Hopper SM90), with instruction encoding, register files, memory layout, and synchronization differing across genera-tions. Table 1 summarizes the key structural differences between SASS and LLVM IR. LLVM IR [ 38 ] is a strongly typed, platform-independent assembly in static single assignment (SSA) form [ 14 ]. It bridges high-level code and machine-specific assembly, with support for x86, ARM, RISC-V, NVPTX, and others. 

## 2.2 SIMT Execution Model 

The NVIDIA GPU architecture is built upon the Single Instruction, Multiple Threads (SIMT) execution model. Threads are grouped into warps of 32 threads that execute in lockstep. When threads within a warp encounter a data-dependent branch, divergence occurs: the warp executes each branch path sequentially, while predicate registers mask off inactive threads. As a result, SASS encodes most conditional behavior through predicated execution rather than explicit branch instructions. Starting with the Volta architecture, NVIDIA introduced Independent Thread Scheduling, which gives each thread its own program counter and call stack, allowing threads to make independent forward progress even after divergence. However, to regain lockstep efficiency once threads reconverge, NVIDIA employs explicit convergence instructions. On Volta and later architectures, BSSY and BSYNC are used; on pre-Volta architectures, SSY and SYNC serve this purpose. 

## 2.3 Predicated SSA and 𝜓 -Functions 

Standard SSA construction [ 14 ] assumes every instruction in a basic block executes unconditionally. Predicated code violates this: two instructions that write the same register under complementary predicates both “execute” in the same block, and standard SSA rename kills the first definition, producing wrong code. Stoutchinin and de Ferrière [ 54 ] introduced the 𝜓 -function ( psi-function ) for IA-64 predicated code: where a 𝜙 -node selects a value based on which control-flow edge was taken, a 𝜓 -function selects based on a predicate register’s value. A predicated write @P0 R0 = x followed by @!P0 R0 = y becomes R0 = 𝜓 (P0 : 𝑥, !P0 : 𝑦 ),semantically equivalent to a select instruction in LLVM IR. This technique is well-studied in the IA-64 compilation literature [ 8, 12 ,58]. Both IA-64 and NVIDIA SASS support predicated execution : al-most every instruction can be guarded by a predicate register (e.g., 

p0 –p63 in IA-64, P0 –P7 in SASS). When the predicate is false, the instruction behaves as a no-op. Both architectures use a unified register file where integer, floating-point, and predicate values are stored; however, the 𝜓 -function concerns only dataflow, not types. To take advantage of the 𝜓 -function, we apply the same principle to the GPU predicated code in Section 5.2. 

## 3 Challenges in GPU Binary Lifting 

CPU binary lifters rely on separate register files, explicit branch in-structions, and publicly documented ISAs [ 7, 24 , 59 ]. SASS provides none of these: its unified register file carries no type information, predication replaces branches for short conditionals, and the ISA is proprietary and changes across GPU generations. Of these lost structures, type recovery is the most critical because it gates IR generation entirely (Section 6.4). We characterize each challenge below, starting with type recovery. 

## 3.1 Type Evidence Is Sparse and Must Be Propagated 

Type recovery is the key technical challenge of GPU binary lift-ing. On CPUs, separate integer and floating-point register files provide type information for free. On SASS, a unified register file holds Int32 , Float32 , Float64 (as register pairs), pointers, packed 

Float16 ×2, and boolean predicates interchangeably. The register name carries no type information. Because LLVM IR requires ex-plicit types on every instruction, a lifter that cannot distinguish 

Float32 from Int32 cannot emit valid IR at all. Most SASS instructions encode type constraints in their opcodes (e.g., FADD operates on floats, ISETP produces a boolean predicate), providing direct type evidence for ∼90% of all instructions (Sec-tion 6.3). However, two challenges remain. First, the remaining 

type-transparent instructions (moves, bitwise logic, and memory operations) pass values without revealing their type, requiring propagation along def-use chains to resolve them. Second, GPU compilation introduces intentional type conflicts : deliberate reinter-pretation of float bits as integers for optimization [ 39 ]. We detect CuLifter: Lifting GPU Binaries to Typed IR 

over 3.8 million such conflicts across our benchmarks (Section 6.3). Incorrect type assignment produces silent numerical corruption: a 32-bit float and an integer with the same bit pattern differ by orders of magnitude but cause no runtime fault. As shown in Listing 1, register R25 is defined by FADD (Float32), consumed by MUFU.RSQ (Float32), then read by IADD3 (Int32), and finally used again by FMUL (Float32). A forward-only approach forces Float32 onto IADD3 , producing wrong integer arithmetic. The correct solution detects the type conflict ( {FP32 } ∩ { Int32 } = ∅)and inserts a bitcast at the boundary. 

Key intuition: GPU type recovery is a propagation problem with conflict detection. The density of type seeds and the length of transparent chains determine how far evidence must travel, and empty intersections reveal implicit type boundaries. We formalize this in Section 4.1 and characterize propagation reach empirically in Section 6.3. 

## 3.2 Control Structure Is Not Explicit 

Control recovery in SASS is fundamentally challenging due to a structural gap in the representation of control flow. LLVM IR ex-pects an explicit control structure in which conditional execution is implemented strictly via branch instructions. In contrast, SASS uses an implicit model in which almost any instruction can be pred-icated. Instead of a conditional branch instruction, SASS models conditional branching via predicate-guarded execution of an uncon-ditional BRA . The actual branch conditions are contained implicitly within the predicate register, whose set-predicate instruction could be many instructions away. In addition, this implicit nature of pred-ication further complicates SSA construction, because a predicated definition updates a value only for the threads whose predicate is true, while the remaining threads implicitly preserve the old value, breaking standard data-flow tracking assumptions. 

Key intuition: control flow in SASS is heavily carried by predi-cate flow. To bridge this structural gap and produce a valid SSA form, the lifter must handle the predication semantics and reconstruct the missing explicit logic from predication correctly. 

## 3.3 Semantic Operations Are Collapsed into Multi-Instruction Patterns 

GPU compilers collapse high-level operations into multi-instruction patterns whose structure changes across GPU generations. A 64-bit subtraction threads a borrow predicate across two IADD3 /IADD3.X 

instructions; if a lifter handles each instruction in isolation, the carry chain breaks and the wrong result is produced. On SM90, a single cp.async in CUDA becomes an orchestrated sequence: 

FENCE.VIEW.ASYNC.S issues a memory fence, LDGSTS.E.128 per-forms a fused global-to-shared copy, and LDGDEPBAR /DEPBAR /BAR. SYNC form an async completion chain. A lifter that treats these inde-pendently would decompose LDGSTS into a synchronous load and store and discard the barrier chain as no-ops. SM90 also introduces packed-conversion instructions such as F2FP.PACK_AB_MERGE_C ,which converts two Float32 values to Float8, saturates, packs both into the upper 16 bits, and merges with a third operand’s lower 16 bits, all in a single opcode. No direct LLVM intrinsic exists; the lifter must synthesize the full decomposition. 

Key intuition: Without pattern recognition, lifted IR is correct in behavior but opaque in intent. We present the full pattern catalog in Section 5.3. 

## 3.4 Implicit Register Dependencies 

SASS instructions may access registers that are not explicitly named in the disassembly. In many cases, an operand denotes only the first register of a contiguous register group, while the remaining regis-ters are determined implicitly by the opcode semantics. For example, 

LDG.E.64 R4, [R2] appears with only two explicit registers, but it actually defines R5:R4 and reads R3:R2 . A conventional lifter that models only the explicitly named operands therefore misses half the data dependencies. This directly affects correctness. If another instruction writes R5 between the load and its consumer, the lifter must detect the conflict, but it cannot, because R5 never appears in the disassembly. Getting this wrong silently corrupts values rather than producing an error. Tensor-core instructions demonstrate an extreme case. On SM75, HMMA.16816.F32 names four registers but actually accesses fourteen. On SM90, HGMMA.64x128x16.F32 ac-cesses 164 registers through four named operands. 

Key intuition: On CPU ISAs, every accessed register is named in the instruction encoding. On SASS, the lifter must reconstruct hidden registers from per-opcode tables before building data de-pendencies; otherwise SSA, def-use analysis, and type propagation all operate on incomplete information. 

## 3.5 Device Functions Are Embedded 

In CPU binaries, every callable function is a distinct ELF symbol with its own entry point, and lifters discover function boundaries directly from the symbol table. On SASS, the compiler embeds device functions inside the kernel function. They are not inlined (they retain their own control flow with CALL and RET instructions), but they share the same SASS section and have no separate symbol table entry.

## 4 Solution Approach 

This section presents our solutions to the two key technical prob-lems at an algorithmic level, independent of SASS-specific details. Section 5 describes the GPU-specific implementation. 

## 4.1 Type Recovery via Constraint Propagation 

We formulate type recovery as a constraint-propagation problem over type sets. Each value 𝑣 carries a set of candidate types T ( 𝑣 ) ⊆ 

𝐿 , where 𝐿 contains Bool; the integer family {Int8 , Int16 , Int32 , Int64 , Int128 }

plus its unsigned counterparts {UInt8 , UInt16 , UInt32 , UInt64 }; and the floating-point family {Float16 , BF16 , TF32 , Float32 , Float64 }. Sec-tion 5.4 formalizes the lattice and width-based partitioning. All values start unconstrained: T ( 𝑣 ) = 𝐿 .

Phase 1: Detect type seed. Instructions with known type se-mantics (e.g., floating-point add ⇒ Float32 , integer compare ⇒

Bool ) narrow their operands’ sets. Most SASS instructions are type-seeding (Section 3.1), but the remaining type-transparent instruc-tions create gaps that propagation must bridge. 

Phase 2: Propagate type information. Type-transparent in-structions (Section 3.1) propagate constraints by intersecting the candidate sets of related operands along def-use edges. A worklist-based fixpoint iterates until no set changes. 

Phase 3: Detect conflicts and insert bitcasts. When inter-section produces ∅, the algorithm has discovered an implicit type boundary , a point where the compiler reinterprets bits across types with no explicit instruction. The conflict resolution rule is: (1) The value retains its definition-site type . (2) The conflicting use site 

receives an explicit bitcast . (3) Once marked conflicted, subse-quent type assignments overwrite rather than intersect, preventing cascading false conflicts. For example, Figure 1 traces the recovery phase through Listing 1. A register holding a float value is read as an integer by a subsequent instruction. Seeding establishes the definition-site type as Float32 .When the integer use requests Int32 , the intersection {Float32 } ∩ {Int32 } = ∅ triggers a conflict: the value keeps Float32 , and the integer use receives a bitcast fp32 to i32 . A later float use of the same register requires no bitcast because the definition-site type is already correct. After convergence, values with multiple remaining candidates are resolved by priority ( Int32 preferred), a conservative choice that preserves bit patterns for a subsequent bitcast . The key quantity is propagation reach : how many transparent hops evidence must travel. We measure this across real GPU kernels in Section 6.3.

## 5 Recovering Analyzable Structure in CuLifter 

This section describes three structure recovery techniques identified in Section 3. 

## 5.1 Recovery Ordering 

The two techniques from Section 4, together with pattern nor-malization, must execute in a specific order as shown in Figure 2: control-flow reconstruction first (to build the CFG with SSA form), semantic operation recovery second (to normalize and to aggregate multi-instruction patterns), and type recovery last (to propagate over the normalized graph). Control-flow reconstruction must come first because both pattern matching and type propagation traverse def-use edges, which require a complete CFG in SSA form. Semantic operation recovery must precede type analysis because the type lattice operates over aggregated values; typing a carry-chain pair before aggregation would mistype the borrow predicate as an in-dependent integer rather than as internal to a 64-bit operation. Stages 1 and 2 are architecture-dependent because instruction en-codings, control codes, and predicate semantics differ across GPU generations.

5.2.2 Implicit Register Expansion. CuLifter expands every instruc-tion to its full set of accessed registers using per-opcode tables. For 64-bit operations, the named register is replaced with an explicit register pair: R4 becomes R5:R4 . For 128-bit operations (LDG.128 , STS.128 ), a register quad R7:R6:R5:R4 is created. A register group is treated as a single SSIR operand with sub-register access, so that type analysis can preserve the type constraint (R3:R2 as an int64 address operand in LDG ) while assigning types to individual registers (e.g., R3 is later used as int32 by IADD3 ). 

5.2.3 Predicate-Aware CFG Construction. We identify basic-block leaders using standard rules (function entry, branch targets, fall-through successors) plus one GPU-specific rule: whenever the pred-icate guard changes, a new block starts. This isolates predicated regions for later analysis. On SASS, conditional control flow is encoded as predicate-guarded execution of unconditional instructions, so the lifter must recover the implicit condition. Unconditional branches become Br termi-nators. For predicated BRA /EXIT patterns, CuLifter extracts the predicate guard and constructs a CondBr with both taken and fall-through successors. For example, a predicated @P0 EXIT terminates only threads satisfying the condition; the remaining threads fall through to the next instruction. A non-SIMT-aware lifter treats this as unconditional termination, making the fall-through unreach-able. SIMT-specific instructions ( BSSY , BSYNC , WARPSYNC ) are kept as ordinary instructions with metadata for later stages. A more complex case arises with dual-predicate branches, where two independent predicates control the same instruction:              

> 1ISETP.EQ.AND P1, PT, R24, 0x1, PT // P1 = (R24 == 1)
> 2ISETP.NE.AND P0, PT, R2, RZ, PT // P0 = (R2 != 0)
> 3@!P0 BRA P1, 0x2810 // if (!P0) branch-on-P1
> 4STL [R1+0x4], RZ // fall-through

Listing 4: Dual-predicate branch: execution guard and branch condition are separate predicates. 

@!P0 BRA P1 has two predicate roles: @!P0 is the SIMT execution guard, and P1 is the branch condition. We represent this as a nested predicated branch; treating either field as primary produces an incorrect CFG. 

5.2.4 Dataflow Graph Construction. On top of the CFG, we build an intra-procedural def-use graph. Nodes are SSIR instructions; edges represent register value flow from definitions to uses. The analysis includes predicate registers and special registers (e.g., SR_TID ) so that later passes can reason about SIMT-specific data flow. Because SSA guarantees each register has exactly one definition, the analysis is a simple two-pass scan: the first pass maps each register to its defining instruction; the second links each use to its definition in constant time.

5.2.6 Device Function Recovery. After CFG construction and be-fore SSA, we recover embedded device functions by identifying disconnected subgraphs unreachable from the entry block. Call targets are resolved through the ELF relocation table. Recovered device functions are either extracted into separate SSIR functions or inlined into the caller. Inlining is the common case: the CUDA compiler inlines most device functions at the PTX level, so the few that survive to SASS are typically small helpers. Extraction is reserved for functions with many call sites (e.g., math library routines compiled as device functions) to avoid code bloat.

## 5.6 Architecture Support 

We support SM75 (Turing), SM90 (Hopper), and SM100/SM120 (Blackwell), with legacy support for SM35 (Kepler) and SM52 (Maxwell). Architecture support is localized to Stages 1 and 2: the parser han-dles different instruction encoding and predicate semantics, and the pattern library adds generation-specific idioms. Type analysis and LLVM generation are architecture independent. Adding a new generation requires extending the parser and pattern library only. The main evolution across generations is the convergence mech-anism. Pre-Volta architectures use reconvergence stacks ( SSY /SYNC ); Turing and later use convergence barriers ( BSSY /BSYNC ). We han-dle both by abstracting convergence as opaque synchronization intrinsics.

## 6.6 Discussion 

Parser completeness. The six function-level lift failures (in CUT-LASS) are caused by unimplemented SM90 instruction variants, pri-marily QGMMA. Adding support requires extending the parser and lowering rules but does not require changes to the type-propagation or control-flow algorithms. Each new GPU generation introduces new instructions, so parser maintenance is an ongoing cost. 

Type recovery is sound but incomplete. Our constraint prop-agation detects conflicts and inserts bitcasts, but it cannot recover types for values that are never consumed by a type-seeding instruc-tion. These values receive the Int32 fallback, which preserves bit patterns but may misrepresent intent. In practice, 1.4% of values remain unresolved (Section 6.3). 

Intra-procedural analysis. Type propagation and control-flow recovery operate within a single kernel function. Device-side func-tion calls (rare in SASS, where the compiler aggressively inlines) are lifted as opaque calls without cross-function type propagation. 

Semantic fidelity. The CPU backend achieves a 73.8% end-to-end pass rate on HeCBench (compile, execute, and bitwise/tolerance-checked output match against the original binary). The remaining 26.2% are kernels that lift cleanly but fail downstream because Jisheng Zhao 1 , Huanzhi Pu 1, Shinnung Jeong, Chihyo Ahn, Hyesoon Kim Georgia Institute of Technology USA 

hardware-specific operations (MUFU approximations, texture sam-pling, SM90 cluster features) have no exact CPU equivalent. The CPU backend is intended for analysis and testing, not performance.

## 7.2 Cross-Platform Execution on x86 and ARM 

Because the lifted output is standard LLVM IR, we can retarget it to non-GPU backends including x86-64 and aarch64 (ARM). We compile lifted IR with clang and execute on commodity hardware, enabling functional testing without a GPU. 

Execution model. The CPU runtime emulates GPU execution for correctness validation, not performance: one OS thread per GPU logical thread, with CTAs executed sequentially. Kernel pa-rameters are passed through constant memory matching SASS’s 

c[0][ARG_BASE + offset] layout, so the lifted function has a 

void() signature with no ABI dependency. The base offset varies across GPU generations (e.g., 0x160 on SM75, 0x210 on SM90, 

0x360 on SM100/SM120) because NVIDIA expands the constant-memory prefix with each generation to accommodate additional system state. CuLifter reads the correct offset from ELF metadata (EIATTR_PARAM_CBANK ) rather than hardcoding it, so the same run-time supports all architectures. 

GPU semantic decomposition. GPU-specific operations have no direct CPU equivalent; we decompose them to portable scalar operations as summarized in Table 11. All decompositions use stan-dard LLVM instructions and C++17 synchronization primitives, with no platform-specific intrinsics or inline assembly.

Numerical fidelity. The hardest semantic gap is NVIDIA’s hard-ware reciprocal ( MUFU.RCP ), which returns an approximation bi-ased by approximately +3 ULPs relative to IEEE 754. The compiler-emitted integer-division sequence depends on this bias: our runtime must bias the IEEE reciprocal result by +3 ULPs to match, or exact IEEE division produces wrong quotients. 

ARM (aarch64) support. Supporting ARM required chang-ing only the TLS model annotation and the LLVM target triple. The entire lifting pipeline (parser, transform passes, type analysis, and IR generation) is target-independent and produces identical LLVM IR regardless of the host CPU. We validated on an ARM Grace Hopper (GH200) system; all failures matched x86 failures with no ARM-specific issue. ARM’s relaxed memory model does not affect correctness because inter-thread communication in the runtime uses std::mutex , which provides full acquire/release semantics on both architectures. The goal of cross-platform execution is analysis and valida-tion, not performance parity. Texture/surface operations and SM90 cluster-level features are not yet supported. 

## 8 Related Work 8.1 CPU Binary Lifters and Their Assumptions 

CPU binary lifters, including McSema [ 19 ], BAP [ 7 ], RetDec [ 34 ], and others [ 9, 13 , 16 , 21 , 33 , 37 ], are mature tools for x86, ARM, and RISC-V. As discussed in Section 3, the assumptions these tools rely on (branch-based control flow, separate register files, documented ISAs) do not hold for SASS. CuLifter: Lifting GPU Binaries to Typed IR 

## 8.2 Type Recovery: CPU vs. GPU 

CPU binary type recovery is well-studied, with two main lines of work. The first is constraint-based inference: TIE [ 36 ] introduced type inference with intersection and union types, Retypd [ 41 ] ad-vanced this direction with recursive types, polymorphism, and sub-typing, and BinSub [ 52 ] reformulates subtyping-based inference us-ing algebraic subtyping, achieving a 63 × runtime improvement over Retypd. The second line applies probabilistic and learning-based techniques: OSPREY [ 61 ] recovers variables and data structures probabilistically, DEBIN [ 27 ] and DIRTY [ 10 ] predict types and variable names from stripped binaries using learning-based models, Manta [ 60 ] combines hybrid-sensitive static analysis with type in-ference for bug detection, and TYGR [ 62 ] infers types for stripped binaries via graph neural networks. All of these systems operate in settings where ISA and ABI structure provide strong type evidence through separate register files, explicit branch instructions, and documented calling conventions. On SASS, that evidence is much weaker because integer, floating-point, pointer, and packed values all share a single architectural register file. GPU type recovery therefore becomes a propagation problem with conflict detection rather than a classification problem. We must also handle intentional type punning, routinely emitted by the GPU compiler for fast reciprocal and exponent manipulation, through explicit bitcast insertion at conflict boundaries. 

## 8.3 Tools for SASS or GPU Binary Analysis 

NVIDIA’s cuObjDump and nvdisasm [ 43 ] provide disassembly and CFG extraction. NVBit [ 57 ] is a dynamic binary instrumentation framework; SASSI [ 53 ] is a compile-time instrumentation frame-work. Ari et al. [ 25 ] demonstrate assembler generation with a Hard-ware Abstraction Layer. Unlike CuLifter, none of these recover analyzable LLVM IR from SASS. Jeanmougin et al. [ 30 ] build CFGs for SASS using abstract in-terpretation to detect divergence. Shoushtary et al. [ 51 ] reverse-engineer 20 SASS control-flow instructions for CFG construction. Sen et al. [ 50 ] apply pattern matching to SASS for memory-access analysis. In contrast, CuLifter translates SASS to typed LLVM IR, enabling compiler-level analysis and retargeting to other architec-tures. 

## 8.4 Running CUDA on Non-NVIDIA Hardware 

Prior work translates CUDA at various abstraction levels [ 3 , 5 , 18 , 22 ,23 , 29 , 31 , 46 , 49 , 55 ]. Source-level tools (MCUDA [ 55 ], Hipify [ 2 ]) convert CUDA source to CPU loops or HIP. COX [ 23 ] and CuP-BoP/SoftCUDA/Volt [ 3 , 22 , 31 ] operate on NVVM IR. Ocelot [ 18 ]lifts PTX to LLVM IR. ZLUDA [ 29 ] translates PTX at runtime. Flex-Grip [ 5 ] reverse-engineers SASS for integer-only FPGA execution. CASS [ 28 ] ports CUDA to AMD at source and assembly level. Unlike all of these, CuLifter targets SASS (the final binary) and recovers typed LLVM IR. 

## 9 Conclusion 

We presented CuLifter, a framework that recovers analyzable LLVM IR from NVIDIA SASS binaries. Our key findings are: (1) Type recovery is the key challenge of GPU binary lifting. SASS’s unified register file erases all type informa-tion. We show that type recovery on GPUs is a constraint-propagation problem with conflict detection. Our ablation confirms this: disabling type recovery drops the x86 pass rate from 73.8% to 0%, a 73.8 percentage-point drop, while disabling other recovery steps has negligible effect. (2) Predicated-to-SSA conversion adapts known techniques. 

We apply 𝜓 -function-based select insertion from the IA-64 literature, extended for SIMT-specific constructs (predicated 

EXIT , dual-predicate branches, convergence barriers). (3) Practical lifting across GPU generations is feasible. 

CuLifter supports SM75 through SM120 and successfully lifts 24,431 of 24,437 GPU functions (99.98%) across eight benchmark suites spanning open-source applications, ven-dor libraries, and optimized ML runtimes, demonstrating that opaque GPU binaries can be brought back into the compiler-analysis stack. The lifted LLVM IR enables downstream analysis (memory access pattern detection) and cross-platform execution on x86 and ARM CPUs.
