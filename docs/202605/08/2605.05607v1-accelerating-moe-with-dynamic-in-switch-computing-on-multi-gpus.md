## 摘要
混合专家模型（MoE）已被许多领先的大模型采用，以降低计算需求。然而，MoE 专家并行（EP）中频繁的 GPU 间通信成为了性能挑战。我们观察到 MoE 中存在大量冗余的 GPU 间数据传输，这些传输有望通过交换机内计算（in-switch computing）来解决。不幸的是，现有的解决方案 NVLink SHARP (NVLS) 仅支持具有规则模式的静态集合通信，无法处理 MoE 中具有不规则模式的动态通信。为了弥补这一功能差距，我们提出了 DySHARP，这是一种用于加速 MoE 的完整动态交换机内计算解决方案，涵盖了通信原语和通信感知调度：1) 动态多内存寻址（Dynamic multimem addressing）协同设计了指令集架构（ISA）、架构和运行时，作为 NVLS 的动态扩展，减少了冗余流量。然而，由此产生的流量减少在两个方向上本质上是不对称的，导致其无法直接转化为加速。2) 以 Token 为中心的算子融合（Token-centric kernel fusion）深度融合了分发-计算-合并（dispatch-computation-combine）流水线，解决了这种不对称性，从而将流量减少转化为实际的加速。与最先进的解决方案相比，DySHARP 实现了高达 1.79 倍的加速。

## Abstract
Mixture-of-Experts (MoE) has been adopted by many leading large models to reduce computational requirements. However, frequent inter-GPU communication in MoE expert parallelism (EP) becomes a performance challenge. We observe substantial redundant inter-GPU data transfers in MoE that can be potentially addressed by in-switch computing. Unfortunately, the existing solution, NVLink SHARP (NVLS), can only support static collectives with regular patterns, incapable of dynamic communication with irregular patterns in MoE. To bridge the functionality gap, we propose DySHARP, an integral dynamic in-switch computing solution to accelerate MoE, encompassing both communication primitives and communication-aware scheduling: 1) Dynamic multimem addressing co-designs ISA, architecture, and runtime, as a dynamic extension to NVLS, reducing redundant traffic. However, the resulting traffic reduction is inherently asymmetric between two directions, preventing it from directly translating into speedup. 2) Token-centric kernel fusion deeply fuses the dispatch-computation-combine pipeline, resolving this asymmetry to translate traffic reduction into actual speedup. Compared with the state-of-the-art solution, DySHARP achieves up to 1.79$\times$ speedup.

### 1. 核心问题与研究动机
*   **核心问题**：在 MoE 架构中，专家并行（EP）导致的 GPU 间通信（分发 Dispatch 和合并 Combine）占据了总执行时间的 50-80%，成为性能瓶颈。
*   **研究动机**：
    *   **通信冗余**：作者观察到 MoE 通信存在严重冗余。分发时，同一 Token 常被发往多个 GPU；合并时，多个 GPU 的输出需聚合回源 GPU。
    *   **现有技术局限**：现有的交换机内计算技术（如 NVIDIA 的 NVLS/SHARP）仅支持**静态**、规则的集合通信（如 AllGather），无法处理 MoE 中**动态**、不规则的通信模式（目标 GPU 集合多变、内存地址不对称）。
    *   **功能鸿沟**：若强行用静态通信模拟动态通信，会产生高达 340% 的无用流量。

### 2. 方法论：DySHARP 核心技术
DySHARP 包含两个关键组件，通过软硬件协同解决上述问题：

*   **动态多内存寻址 (Dynamic Multimem Addressing)**：
    *   **核心思想**：扩展 NVLS 的寻址机制，使其支持动态目标列表和不对称内存布局。
    *   **技术细节**：
        *   **指令集与数据包扩展**：设计了新的 `dymultimem` 指令，数据包携带一个“代数索引”和“目标专家列表”。
        *   **硬件内存管理器**：在 GPU 的 Hub 层引入硬件映射表（AL Table）和缓存（AL TLB），将逻辑上的代数索引动态映射到各 GPU 本地实际的物理布局索引，实现“一发多收”和“多收一减”。
*   **以 Token 为中心的算子融合 (Token-centric Kernel Fusion)**：
    *   **核心思想**：解决交换机内计算带来的流量不对称问题（分发减小了上行流量，合并减小了下行流量，导致带宽利用不均）。
    *   **技术细节**：
        *   **流水线并行**：打破算子边界，在 Token/Tile 粒度上将分发、GEMM 计算和合并操作深度融合。
        *   **Token 追踪器 (Token Tracker)**：通过硬件表（TS Table, OR Table）实时监控每个 Token 的就绪状态，一旦 Token 完成计算立即触发合并通信，从而让分发和合并在时间上重叠，平衡双向带宽。
