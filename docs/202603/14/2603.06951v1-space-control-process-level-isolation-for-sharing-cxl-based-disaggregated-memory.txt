## ABSTRACT 

Memory disaggregation via Compute Express Link (CXL) enables multiple hosts to share remote memory, improving utilization for data-intensive workloads. Today, virtual memory enables process-level isolation on a host and CXL enables host-level isolation. This creates a critical security gap: the absence of process-level memory isolation in shared disaggregated memory. We present Space-Control, a hardware-software co-design that provides fine-grained, process-level isolation for shared disaggre-gated memory. Space-Control authenticates execution context in the hardware and enforces access control on every memory access and amortizes lookup times with a small cache. Our design allows up to 127 processes running concurrently on 255 hosts to share memory with only 1.56% storage overhead. In a gem5 + Structural Simulation Toolkit (SST) based CXL model, Space-Control incurs minimal performance overhead of 3.3%, making shared disaggre-gated memory isolation practical. 

## 1 INTRODUCTION 

The growing demand for memory in modern data-intensive applica-tions has driven the industry toward memory disaggregation [ 21 ]. Disaggregation decouples compute and memory resources and con-nects them via high-speed fabrics. Standards such as the Compute Express Link (CXL) [ 2, 3, 29 , 55 ] enable multiple systems, called 

hosts , to access memory on a remote memory-only device . This improves utilization and reduces redundancy [ 43 ]. The CXL 3.0 [ 2 ]standard supports memory sharing at the cache-line granularity across hosts for the first time, making it a promising foundation for large-scale deployments. Current CXL-based sharing mechanisms provide only coarse-grained, host-level access control: once a host is authorized, all processes on that host can access the shared region. This violates the principle of least privilege [40 ]. The all-or-nothing model leaves intra-host isolation to the operating system (OS). Since memory is shared across multiple hosts, there is no cross-host user or process visibility. Stark et al. [57 ] call the coarse-grained sharing in CXL as 

“sledgehammer-level protections.” When coupled with a compromised kernel [ 16 , 23 , 35 , 37 ], this gap can escalate privileges and access sensitive data, creating a significant security risk [38]. Before discussing the problem in details, we first need to consider the characteristics of shared memory workloads. Disaggregated memory workloads are an active research topic [ 25 , 46 ]. These worloads partition their data across local and remote memory. Al-though multiple hosts may map the same physical remote memory region via CXL, they are not necessarily accessing the same data within that region. Therefore, for isolation, we need to guarantee 

fine-grained permissions for user-defined arbitrary memory ranges for the remote memory. Tigon [ 25 ], a CXL-based key-value store software, optimizes fine-grained data movement and resource utilization in the remote memory across hosts. The authors point out a lack of access control in CXL and implicitly assumed the existence of fine-grained permis-sions to enforce isolation across shared regions. Beyond key-value stores, graph analytics is an another example where users share graphs or sub-graphs in the remote memory across hosts, each running a graph kernel. This requires fine-grained authorization such that users on a host can read or update only its assigned parti-tions ( e.g. , specific compressed sparse row slices or property arrays) without exposing the rest of the graph. Other examples include the sharing of machine learning model weights (especially in ex-pert models [ 49 ]) across hosts to improve utilization and reduce redundancy. Today, systems enforce such policies either at the OS level or at the software level, without hardware-backed guarantees. This motivates us towards researching hardware-enforced isolation that remains effective even if the OS is compromised. In principle, we need guarantees akin to confidential computing, namely, creating an enclave for isolation of the address space and independence from the OS. In practice however, current Trusted Execution En-vironments (TEEs) are ill-suited to shared disaggregated memory as commercial TEE designs like Intel SGX [ 7], Intel TDX [ 9 ], AMD SEV [ 33 ], CXL-TEE [ 2, 3 ] etc. cannot share memory between en-claves [ 64 , 65 ]. Today’s TEEs have security monitors that operate only within a single host, with no cross-host coordination, making it challenging for existing TEE designs to securely share remote memory across hosts. Furthermore, they introduce notable perfor-mance overheads, reduce deployment flexibility, and expand the trusted computing base (TCB) [ 12 ]. This leads us to the central research question in this paper: “How can we ensure isolation in shared disaggregated memory that enforces the principle of least privilege and operates independently of the OS with minimal overhead?”

In this paper, we present a hardware-backed access control frame-work that enforces isolation defined by a process . A process is the smallest OS abstraction that issues user-mode loads/stores (LD/ST) under its own distinct address space. Further, our design ensures compatibility with the virtual memory infrastructure. Across the ecosystem, existing access control mechanisms fail to provide the necessary granularity and trust guarantees for disag-gregated environments. The POSIX shared memory API [ 5 ] solved inter-process shared memory on a single host. However, cross-host usergroups may differ, so POSIX shared memory does not address cross-host sharing. Software-only approaches such as namespaces 

and cgroups [1 ] offer isolation, but requires trusting the OS for cross-host visibility. Hardware-only solutions like DeACT [ 38 ] and the CXL specification support coarse-grained sharing (1 GiB and 256 MiB respectively), limiting flexibility. Prior proposals based on capabilities, which guarantee isolation, argue for finer access-control at the object-level for CXL fabrics [ 57 ]. However, capabil-ity systems like CHERI demand pervasive CPU/ISA/compiler/OS changes [18, 59, 63]. The core difficulty is balancing fine-grained authorization with scalability and efficiency. Prior works have emphasized horizontal 

access-control, i.e. host-level permissions [ 2 , 38 ], whereas disaggre-gated memory also needs vertical access-control, i.e. per-process or per-virtual machine across hosts. Tagging every 4 KiB page of a 1 TiB memory region with permissions (2 bits) per host (up to 256) and per process or per-virtual machine (up to 128) permissions re-quires 2 TiB of metadata (200% overhead). Therefore, naïve lookup table designs become infeasible, and permission checks risk becom-ing performance bottlenecks. Prior research like Mondrian [ 61 ]trades off storage overhead for higher lookup latency on a single host using sorted tables to solve this problem. However, in shared disaggregated memory (SDM) each host’s permission table must be replicated for cross-host visibility. Therefore, we require a solution that provides shared memory isolation without incurring prohibitive metadata or performance overhead. We present Space-Control, a scalable access-control frame-work for isolation in shared disaggregated memory. Space-Control registers and authenticates address spaces defined by a user pro-cess at the host using a secure process attribute context engine (SPACE), instead of the OS. Further, we add a permission checker to enforces access-control on every LD/ST at the point of egress. Finally, we leverage the under-utilized fabric manager (FM) in CXL for managing cryptographic keys and permissions. Access control metadata is co-designed with the kernel to achieve scalability. This reduces the total storage overhead of prior ap-proaches from 200% to a fixed 1.56% of the total memory capac-ity when 255 hosts share memory of 4 KiB pages across 127 pro-cess on each host. Our design delivers hardware-backed isolation with space-efficient metadata, addressing both the flexibility and granularity requirements of the remote memory. Evaluation on GAPBS [17] shows a marginal 3.3% performance overhead. The contributions of this paper are: 

• We identify a critical isolation gap in shared disaggregated memory (SDM). While current systems provide process-level isolation (via the OS) and host-level isolation (via CXL), they lack process-level isolation for SDM. 

• We propose Space-Control, a system providing hardware-enforced, process-level isolation for SDM that remains con-fidential even if the OS is compromised. This architecture includes a hardware validation module and an access-control permission checker. 

• We address metadata overhead through a hardware-software co-designed permissions management technique. 

• We present the hardware design of Space-Control and pro-vide a quantitative evaluation of its performance and space overheads. The rest of the paper is organized as follows. §2 defines the prob-lem and §3 explains the background. §4 describes our solution and §5 discusses the implications of the solution. The following sec-tions discusses the evaluation methodology and results. §8 explains related works. The work is concluded in §9. 

## 2 PROBLEM DEFINITION 2.1 Threat Model 

Principle and Scope. Space-Control’s trust model follows the prin-ciple of least privilege [ 40 , 52 ]. Only address spaces defined by a process, that require access to the remote memory, should be al-lowed to access the same. We provide confidentiality to local and remote pages of such process’ data. Further, we provide integrity 

to all remote pages and detect any intrusion by the OS. 

System setting. We consider a HPC-like deployment where sev-eral multi-tenant hosts share SDM via CXL. Users directly interact with the host OS via user processes. Applications interact with SDM through user processes that map regions ( e.g. , DAX). Each process has both local (on-host) pages and remote (SDM) pages. We do not support virtual machines. 

Trusted. We trust the user process (its code and keys) that is granted SDM access, and the additional hardware we propose. We assume on-chip and CXL hardware to function correctly. 

Untrusted. The OS (including the kernel) is untrusted in the ker-nel mode (or S-mode ). Our only kernel change is a narrow extension that enables the OS to understand process contexts and SDM ranges; it contains no authorization logic. If this change is absent, SDM remains unmapped and the hardware permission checker rejects all SDM requests. Other user or kernel processes on the host may col-lude with the OS to break confidentiality. Off-chip components such as local DRAM modules and other external devices are untrusted. 

Attacker Capabilities. We consider a powerful adversary capable of controlling the OS. OS-level vulnerabilities can escalate privi-leges and override permissions [ 23 , 37 , 38 , 44 ]. The attacker can manipulate page tables, virtual memory mappings, replay or forge identity, and invoke privileged instructions. Physical attacks, denial-of-service and side-channels on the hardware are out of scope.

## 3 BACKGROUND AND CURRENT APPROACHES 

This section explains access-control on a single-host system and describes the missing gap in disaggregated memory systems. Table 1 summarizes the related works in terms of the four requirements mentioned in §2.2. On a single-host system, the virtual memory manages per-process page tables. The hardware (TLB/MMU) enforces translation and per-page W/R/X permissions [ 15 ]. A process cannot access another’s pages unless they are explicitly shared. Address space identifier (ASID; ARM) or process context identifier (PCID; X86) tags TLB entries to distinguish address spaces and reduce shootdowns. They aid performance but do not provide authorization. Intel’s PASID/SVA [ 4 , 27 ] binds I/O accelerators to a process’ vir-tual address space for device-initiated DMA via the IOMMU. This is orthogonal to CPU-originated LD/ST via DAX . OS-managed mecha-nisms such as Border-Control [ 48 ], MPK [ 50 ] and ARM MTE [ 14 ]provide intra-host memory protection but assume a trusted ker-nel. Mondrian memory protection defines domains as a set of pro-cesses with identical permissions, defined by the OS. These domains are sorted and stored as separate entries on a permission table. TEEs [ 6, 7, 9 , 33 ] provide isolation and attestation, but are ill-suited for sharing across hosts [2, 3] and add significant overhead [12]. Disaggregated memory [ 21 ] decouples compute from memory and lets hosts map remote ranges over a fabric [ 10 ]. The fabric manager (FM) manages the set of hosts sharing memory on a set of devices. Prior efforts include OpenCAPI [ 26 ] and Gen-Z [ 28 ], but current deployments center on CXL [ 2 , 3, 29 , 43 , 55 ]. The critical difference for our work is the translation and authorization path .Gen-Z introduced a host-side zMMU and two-stage translation (virtual address (VA) →remote virtual address →physical address (PA)), which enabled designs like DeACT [ 38 ] to decouple transla-tion from access-control. In contrast, CXL removes the host-side zMMU and reuses the CPU’s TLB to access remote memory directly. With CXL 3.0, multiple hosts may share the same physical memory range [ 2] (called sharing ). However, authorization is host-level and coarse (dynamic capacity region granularity) under the FM. In prac-tice, to avoid the OS zeroing-out the bits, an SDM range is mapped as a direct access (DAX) region, similar to the heterogeneous pro-gramming model [30, 46]. Prior works [ 18 , 38 ] either target the Gen-Z model or rely on a trusted OS for isolation [ 54 ]. CHERI-style capability [ 18 , 59 , 63 ]systems offer principled, fine-grained authority tracking [ 57 ], but adoption requires ISA/compiler/OS changes across all hosts. CHERI uses 128 bit pointers to maintain capability per host, maintaining the metadata across 256 hosts take 12.5% more storage ( 256 × 1𝑏 per 16 Bytes). In addition, programs needs to be rewritten to understand 

capabilities-aware LD/ST instructions [ 63 ]. We instead design a process-level authorization path compatible with existing virtual memory infrastructure and CXL deployments. 

## 4 SPACE-CONTROL 

Space-Control enforces per-process authorization (R1) for shared disaggregated memory outside the OS’ control (R2). We achieve this by (i) authenticating the currently executing process, (ii) tagging its memory requests with a hardware-backed process identifier, and, 

(iii) validating each SDM access against a permission table. Space-Control introduces three key components: (1) Secure Process Attribute Context Engine (SPACE): A light-weight hardware module that authenticates processes using hardware-rooted identity, defined by the hardware process context , i.e. reserved PCID/ASID, and, the pointer to the base of the page table ( CR3/SATP/TTBR ); referred to as BASE_P .The engine monitors context switches and ensures that only verified contexts can access the SDM. It offers minimal MMIO interface to the programmer. In addition, it has logic for gen-erating cryptographic labels, and, a counter. (2) Permission Table: An array of permission entries that maps PA ranges; shared or exclusive; of the SDM to the set of authorized contexts. Entries can be dynamically updated to satisfy R4. (3) Permission Checker: An on-chip hardware unit placed after the last-level cache, before the local DRAM controller and the CXL downstream port, responsible for enforcing access control on each LD/ST. It is integrated into the memory access pipeline that actively validates each SDM request against the permission table .Space-Control makes a key observation in existing works to, i.e. 

access control metadata increases to maintain each host’s processes that are allowed to access the remote memory (estimated to be 200% in the naïve approach; §1, 12.5% in CHERI; §3). Therefore, we opt for a hardware-software co-design, explained in §4.2.2, to satisfy R3. We minimally modify the kernel to reserve contiguous process context IDs (ASID/PCIDs (referred to a HWPID henceforth) instead of Linux PID) in the kernel for SDM processes. These HWPIDs are assigned by the SPACE, instead of the OS. The OS still remains untrusted as our correctness never depends on kernel behavior. ## 4.1 Principle of Operation 

Now that we have given an overview of the design, we explain the work flow in three distinct phases: (a) process creation, (b) runtime protection, and, (c) dynamic updates. 

4.1.1 Process Creation. Figure 2 illustrates the process creation and permission request workflow in Space-Control. Space-Control scopes to protect user processes to satisfy R1. The user registers their process as trusted via SPACE hardware instead of the OS, and, creates an entry in the permission table to ensure permission enforcement. Action 1 requests a range of memory to be mapped into a trusted process’ address space, running on the 𝑖 𝑡ℎ host. The user is responsible for requesting the right permissions. This can be achieved either using a middleware or an upper layer protocol as mentioned by prior works [30]. SPACE maintains a list of free HWPIDs assigned via an MMIO doorbell ( GET_NEXT_PID() ). The system-level driver assigns a HW-PID from the free list. This breaks the dependence on the OS for a process identifier (R2). The permission table that maintains each permission entry is stored in the SDM. A new entry ( entry_t ) is written into the proposed update section of the permission table directly (Action 2 ). The FM is automatically notified about the update, and decides whether to approve the request. If the entry is approved, the FM commits the entry (Action 3 ) into the table. The FM immediately generates a cryptographic public label ( 𝐿 𝑒𝑥𝑝 ) using its private key (K FM ) for authentication and stores it into the permission table (Action 5 ). The FM automatically optimizes the permission entry if entries’ ranges overlap. In addition, it responds to the host with the public label (Action 6 ), which is intercepted by SPACE for cryptographic label comparison. The user process is now assumed to be trusted and the execution continues.

Therefore, we need to validate our trusted process at every context switch to maintain R1. Figure 3 depicts the runtime protections provided by Space-Control. 4.1.3 Revocation and Update Propagation. We rely on CXL’s back invalidates (issued via BISnp ) to propagate updates to the permis-sion table. When an update is committed to the table by the FM, every host automatically receives a BISnp , ensuring that the permis-sion is invalid even if the entry is cached in the permission cache. The FM updates the 𝐿 𝑒𝑥𝑝 for the same host, ensuring global trans-parency and satisfying R4 of dynamically updating permissions. The driver is responsible for calling SPACE doorbells for cleaning up HWPIDs. Permission entries with no hosts are automatically cleaned up by the FM.

4.2.3 Permission Checker. The permission checker, shown in Fig-ure 6 is designed in tandem with the previous components. It is placed on-chip of a host, after the last-level cache. Every LD/ST of a trusted process has the A-bits tagged on to the PA. 𝐻𝑊 𝑃𝐼𝐷 𝑙𝑜𝑐𝑎𝑙 ,

i.e. the list of trusted processes on a host, is implemented as a bit vector, and stored in the permission checker. The checker validates whether the trusted process’ tagged addresses have access rights to a given PA, thus maintaining R1 and R2 at all times. At runtime, every SDM LD/ST is associated with a corresponding permission entry in the table that encodes the range and permission attributes. The checker fetches the same entry from the permission table. Homonyms never appear in our design as PA are used for range checks. The lookup latency is proportional to the number of permission entries. To amortize the lookup latency and avoid round-trips on every remote memory reference, we add a small fully-associative permission cache. To track each outgoing request and its corresponding permission, we use a limited number of permission status holding registers. Permissions are enforced (permission attributes and HWPIDs) at the response-end, where the response of a remote memory request is stalled via buffering until all its corresponding permission responses arrive. This ensures all LD/ST are committed in-order. Finally, we ensure the confidentiality of local pages. A trusted process’ local memory addresses will have the A-bits tagged. We add a memory encryption engine for encrypting these pages us-ing the host’s private key ( 𝐾 ℎ𝑜𝑠𝑡 ). The design therefore provides confidentiality for a trusted process’s local data. 

4.2.4 Extensions to the Fabric Manager (FM). The FM serves as the trusted coordination point for Space-Control. Today, the only role 

> 6

of the FM is to set up the hardware tables for access control (§3). We extend its role to manage the permission table in the SDM, and, generate public labels for context authentication for each host. Upon granting access to a process on a host, the FM issues a public label ( 𝐿 𝑒𝑥𝑝 ) that cryptographically binds a context ( HWPID ,

BASE_P ) to its permitted memory ranges. The label is stored in the label section of the metadata and the entry, which has a defined range, is committed to the permission table. In addition, the FM is also responsible for optimizing the permission table to minimize the total number of entries. 

## 5 DISCUSSION 

In this section, we show the security analysis to validate our claims and justify our design choices. 

## 5.1 Security Analysis 

Our claim is this paper is that we guarantee the confidentiality of trusted context’s data and trust the disaggregation hardware for the availability .

5.1.1 Malicious host processes. The host can spawn multiple pro-cesses, which tries to access the SDM. Space-Control validates the trusted process’ context via the SPACE. The context ( HWPID , BASE_P )cannot be forged by any other process. Any LD/ST generated by a trusted context is further checked by the permission checker. The checker raises a fault when a non-trusted process (without the tagged A-bits ) tries to access the remote memory region directly. 

5.1.2 Compromised system software. We assume a strong threat model in which the OS may collude with untrusted processes to violate the confidentiality of a trusted process’s local and remote data. We therefore use hardware-defined context ( HWPID , BASE_P ). This prevents the OS from forging context labels. Although the OS can still modify page tables, such manipulation does not compromise remote memory. If the kernel maps the SDM region into an untrusted process’s address space, SPACE rejects the access as authentication fails, and the permission checker blocks any unauthorized LD/ST. A stronger threat arises when the kernel attempts to compromise the local memory confidentiality of a trusted process. Since local memory is untrusted in our model, the kernel may remap the page tables of a trusted process and a malicious process to the same physical memory region. The trusted process remains unaware of the remapping and continues to use its local pages, while the malicious process gains read access to those same pages, violating 

confidentiality .To prevent this attack, Space-Control employs a memory en-cryption engine that encrypts all local memory pages belonging to trusted contexts, identified by their tagged A-bits . This ensures 

confidentiality even if the OS aliases page tables. However, while en-crypted data prevents unauthorized reads, a malicious process could still overwrite ciphertext, affecting integrity . When such writes are detected, Space-Control identifies the presence of compromised system software, satisfying our target threat model from §2.1. 

5.1.3 Malicious DMA devices. Space-Control assumes that SDM is accessed primarily via CPU LD/ST operations using /dev/dax [ 46 ]. Unlike system DRAM, a DAX region is not part of the system mem-ory map. It is exposed as a character device and mapped into a processes’ virtual address space. Consequently, DMA-capable de-vices, such as GPUs and other accelerators, cannot directly access this region without explicit peer-to-peer mappings. This distinction reinforces the gap we target, i.e. , existing PASID-based Shared Vir-tual Addressing (SVA) mechanisms only apply to device-initiated DMA and do not cover CPU-originated accesses to SDM. Therefore, hardware enforcement at the CPU egress point remains essential for guaranteeing process-level isolation in scenarios where the OS cannot be trusted. 

## 5.2 Design Choices and Limitations 

Space-Control introduces additional wiring overhead by tagging PA with the A-bits . The design is inspired by the C-Bit of AMDSEV [ 33 ]. We justify our design choice of extending the PA as today’s state-of-the-art architectures uses up to 57 out of 64 bits [ 8, 22 ]. This allows flexibility of using the remaining unused bits to tag the address space. Intel hardware already adds support to extend the upper 7 bits of the physical addresses in the kernel using Linear Address Masking [ 31 ]. We allow up to 127 processes per host to ensure 64 bits address compatibility. For simplicity, we choose to place the HWPID tag bits into the most significant bits of the address, which does not cause conflicts with the existing wiring for the TLB, the caches and ensures CXL compatibility. An alternate method of protecting memory was via a reverse mapping table ( 𝑃𝐴 → ( 𝑉 𝐴, 𝐻𝑊 𝑃𝐼𝐷 )) per host, similar to AMD-SEV [ 33 ]. Such tables are also popular to identify hot cache lines for page migration in CXL-related prior works [ 42 ]. A reverse mapping from PA to (VA, PID) allows the hardware to infer which process owns a given physical page without modifying the address format. However, given the current programming model for SDM, the base pointer to the mmap region is fixed and addresses are accessed based on offsets. We chose to adopt a simple permission table where the mapping from PA to (host, ( HWPID , BASE_P )) is enough to ensure that the OS cannot modify the page table to remap the SDM. Based on our threat model, our design guarantees confidentiality 

to the data at all times and integrity to the data in the SDM. However, we rely on the CXL hardware to function correctly and provide 

availability . This distinguishes Space-Control from a TEE. If the trusted process is compromised at runtime, Space-Control fails both 

confidentiality and integrity guarantees.

There is a pronounced elbow in the cache-benefit curve: a modest permission cache delivers most of the gain. With just 2 KiB of cache, the hit rate reaches 99.9% and the CPI overhead improves by 2.3 ×

over the uncached Space-Control configuration on GAPBS. The sharp drop in miss ratio from 1 KiB to 2 KiB stems from the way the sorted permission table is searched: a binary search repeatedly revisits a small set of internal decision nodes across lookups, while the leaf (the final permission entry for the specific range) is the part most likely to be uncached. For a 16 GiB shared memory, the binary search touches at most 24 nodes at worst. A 2 KiB cache holds 32 entries, which exceeds the depth and keeps most internal nodes resident. Only the leaf entries tends to miss. This scaling intuition generalizes. For 1 TiB shared memory (up to 𝑙𝑔 (240 ) = 40 probes), 4 KiB cache (64 entries) exceeds the worst-case search depth, amortizing the lookup cost. This makes cache auto-tuning straightforward, i.e. choose a capacity that meets or slightly exceeds 𝑙𝑔 (table size ).Compared to the CXL baseline, a 16 KiB cache leaves only 3.3% marginal performance overhead. Consistent with our previous re-sults, caching also trims enforcement stalls, and benefits show di-minishing returns beyond 4 KiB in the worst-case fragmentation for most workloads. A small permission cache converts Space-Control from lookup-bound and stall-heavy to data-dominant, aligning with the design flow we established so far. 

7.1.7 Permission Revocation. Updating permissions follows the CXL’s back invalidate snoop ( BISnp ) protocol [ 2]. When a host updates the permission table via a new entry and the entry is approved and pushed into the table by the FM, a CXL BISnp is sent on the network. Permission caches on hosts caching that entry will automatically invalidate the permission entry from its own cache. The latency of performing a permission revocation is the same as CXL’s BISnp latency. One limitation of permission revocation is that it fails to pre-vent on-flight instructions getting committed. While Space-Control ensures response-side permission checks, meaning that access con-trol is done after receiving responses instead of apriori approvals, there still exists a tiny window where an attacker can read (not write) data from the SDM. This problem however is the CXL’s cache coherence problem [2, 30], which is orthogonal to our work. 

## 7.2 Hardware Overhead

This implies that in the worst case where every 4 KiB page of the SDM will have a 64 Byte entry, resulting in a maximum of 1.56% overhead.

## 8 RELATED WORKS 

Memory disaggregation. §3 gives a brief introduction to disag-gregated memory. Prior works like DeACT [ 38 ] relies on Gen-Z’s host-side zMMU [ 28 ] and two-stage translation to decouple address translation from permission checks. CXL removes the host zMMU and reuses the CPU’s TLB [ 2], so Space-Control enforces autho-rization after the last-level cache. RDMA-based works like [ 58 ] is orthogonal to CXL. 

OS-managed mechanisms. OS-managed protections via namespaces 

and cgroups [1 ], MPK and ARM MTE provide intra-host protec-tion but assume a trusted kernel. LegoOS scales disaggregation but also trusts the OS for isolation. Space-Control removes the OS from the enforcement path (§3). [ 14 , 50 , 54 ] Border Control [ 48 ]sandboxes accelerators with page-level checks under a trusted OS. Space-Control is inspired by this work and builds on this insight by treating untrusted hosts analogous to untrusted accelerators, enforcing access control through a hardware permission checker integrated into the memory access path. 

Domain/capability designs. Mondrian [ 61 , 62 ] introduces the con-cept of domains and uses a sorted protection table to trade-off stor-age for higher lookup latency. Space-Control adopts the sorted-table idea but enforces at the CPU egress with process-level isolation and an untrusted OS (§4). CHERI-style capability systems offer princi-pled fine-grained authority but require pervasive ISA/compiler/OS changes [ 57 , 59 , 63 ]. Bresniker et al. [ 18 ] proposed the addition of memory-side capability check to extend CHERI to the disaggregated memory space. The memory-side capability check is for address 

> 11

range check per process per host, creating a high metadata over-head and restricting scalability. Furthermore, it requires a trusted OS for process coordination. We target deployability on existing CPUs and CXL fabrics using simple techniques like LD_PRELOAD .

TEEs and enclave models. TEEs [ 7 , 9, 33 , 65 ] provide isolation and attestation but are ill-suited for multi-host shared ranges [ 2 , 3] and add non-trivial overhead [ 12 ]. Sharing memory enclaves on a sin-gle host is an active research topic [ 64 , 65 ], however commercially available TEEs are yet to adopt it. Minerva [ 13 ] is an extension to the TEE design for disaggregated memory, where the authors made a fault-tolerant metadata management scheme for disaggregated memory. ShieldCXL [ 19 ] proposes an oblivious TEE for CXL sys-tems. Li et al. [ 41 ] proposes an efficient technique to encrypt CXL flits for designing a TEE on a CXL system. Space-Control enforces authorization without enclaves. 

## 9 CONCLUSION 

This work addresses a critical gap in disaggregated memory sys-tems: the lack of process-level isolation in shared disaggregated memory. Space-Control provides a scalable hardware-software co-design that enforces fine-grained access control without relying on the OS. The design remains fully compatible with existing virtual memory abstractions. Our design uses a space-efficient metadata structure and leverages the CXL fabric manager to scale across many hosts and processes, incurring only 1.56% memory overhead and 3.3% performance overhead. By reconciling strong security with system scalability, Space-Control establishes a practical foundation for secure disaggregated memory. For future work, we plan on extending Space-Control into the TEE realm with shared enclaves across hosts, and providing availability at all times.

## A ADDRESS TRANSLATION AND ACCESS CONTROL IN CXL 

CXL defines hosts and devices . Hosts are compute nodes with or without local memory. The devices on the other hand, are mem-ory devices. There is a global address space, where devices are mapped into. Hosts may choose to map a part of their own address space into the global address space. The global address space is one contiguous address space. Until CXL 2.0, there was no notion of sharing memory ranges among hosts. To facilitate memory sharing, additional hardware is added. The fabric manager (FM) is respon-sible for binding hosts and devices to the global address space. A Global Fabric Attached Memory (G-FAM) is a collection of hosts and devices in one single contiguous global physical address space. CXL switches are responsible for routing a given request to its corresponding device. Hosts can have both local and remote memory ranges. Local ranges are not exposed to the G-FAM, therefore, not appearing in the global address space. The remote memory addresses reside outside the host. Note that both of these addresses are physical addresses. To distinguish between host’s physical addresses and a device’s physical address, the terminology defines host physical address (HPA) and device physical address (DPA). A host is assigned a source port ID (SPID) and a device is assigned a device port ID (DPID). New tables are added to the CXL switch and the G-FAM device to enable sharing. Fabric address segment table (FAST) and the Interleave DPID table (IDT) are added to a CXL switch. On the other hand, G-FAM devices (GFD) decoder table and the SPID Access Table (SAT) are added to the G-FAM device. In CXL terminology, devices are referred to as dynamic capacity devices (DCD). A DCD is divided into multiple regions called DCD region of size 256 MiB [ 2]. Access permissions for a host is determined at the granularity of a DCD region. Figure 15 shows the journey of a virtual address generated at a host to its corresponding device. Action 1 in the figure shows the generation of virtual address (VA) at the host. The host translates this VA to a host physical address (HPA) using the host’s MMU/TLB 

> 13

and send the HPA as a request address on a CXL flit to the outgoing CXL downstream port. The first component a CXL flit encounters is a CXL router. The router is responsible for appending the SPID and the DPID of the address into the request flit. Action 3 shows that forwarding of the request flit to the FAST table to determine the DPID. The request is further sent to the IDT to determine if the address is interleaved into multiple GFDs. If a valid destination does not exist for the DPID, the host is notified and the flit is dropped. The appended request can be seen in Action 4 . The CXL router then forwards this request to the GFD (Action 5 ). The tables for one set of G-FAM is responsible for checking access permission for a given address on a flit for the corresponding DPID and the SPID. The incoming request flit address is looked up in the GFD table (Action 6 ). Once the correct GFD is decoded, the request is then forwarded to the SAT. The GFD is also responsible for translating the HPA to its corresponding device physical address (DPA). The SAT table hosts access permission for a memory group. The table is querried with the SPID to determine whether the given host has permissions for a given memory group. In case the flit is a write on a HPA, a Back-Invalidate Snoop (BISnp) is sent back to all the hosts sharing that HPA (Action 7 ).
