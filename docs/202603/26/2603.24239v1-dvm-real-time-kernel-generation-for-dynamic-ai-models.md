## 摘要
动态性在 AI 计算中非常普遍，例如模型中的动态张量形状和动态控制流。由于编译时间较长，现有的运行时编译会损害模型效率，而离线编译器为了覆盖动态模型所有可能的执行实例，要么面临漫长的编译时间和巨大的设备内存占用，要么为了易用性而牺牲了优化机会。在本文中，我们重新思考了动态模型运行时编译的可行性，并指出其成功的关键在于加速编译过程或隐藏编译开销。为此，我们提出了一种实时编译器 DVM。在 DVM 中，我们设计了一个基于字节码虚拟机的运行时算子编译器，能够根据输入为每个动态算子实例进行高效且有效的编译。具体而言，我们不再将程序编译为机器码，而是在 CPU 上将算子程序编码为字节码，并在 NPU 上将字节码解码为虚拟指令以直接执行。基于该运行时算子编译器，我们进一步提出了一种算子融合器，它在静态图上执行基于符号推导的融合，在动态图上执行运行时融合。该融合器支持基于模式和基于堆叠的融合，以增加融合机会。对算子、子图和模型的评估表明，与 TorchInductor、PyTorch-eager 和 MindSpore-graph-O0 相比，我们在算子/模型效率方面提升高达 11.77 倍，在最大编译时间方面快了多达 5 个数量级。

## Abstract
Dynamism is common in AI computation, e.g., the dynamic tensor shapes and the dynamic control flows in models. Due to the long compilation time, existing runtime compilation damages the model efficiency, while the offline compilers either suffer from the long compilation time and device memory footprint to cover all the possible execution instances of a dynamic model, or sacrifice optimization opportunities for usability. In this paper, we rethink the feasibility of runtime compilation for dynamic models and identify that the key for it to work is to speed up the compilation or hide the compilation overhead. To do this, we propose a real-time compiler, DVM. In DVM, we design a runtime operator compiler based on a bytecode virtual machine to perform effective and efficient compilation for each dynamic operator instance given its input. Specifically, instead of compiling programs into machine code, we encode the operator program into bytecode on the CPU and decode the bytecode into virtual instructions for direct execution on the NPU. Based on the runtime operator compiler, we further propose an operator fuser, which performs symbol-deduction-based fusion on static graphs and runtime fusion on dynamic graphs. Both pattern- and stacking-based fusion are supported to increase fusion opportunities. Evaluation on operators, subgraphs, and models shows that, compared with TorchInductor, PyTorch-eager and MindSpore-graph-O0, we are up to 11.77$\times$ better in terms of the operator/model efficiency and up to 5 orders of magnitude faster in terms of the maximum compilation time.

*   **核心问题**：AI 模型中的动态性（如 LLM 的动态序列长度、动态控制流）给编译器带来了巨大挑战。
*   **研究背景**：
    *   **运行时编译（JIT）**：如 TorchInductor，编译时间极长（比算子执行时间慢 $10^5$ 倍），导致严重的运行抖动。
    *   **离线编译**：难以覆盖所有可能的形状组合，且为了通用性往往牺牲了算子融合等优化机会。
    *   **现有方案局限**：分桶（Bucketing）和微内核（Micro-kernels）在面对高维动态形状或复杂融合算子时，存在内存占用大或搜索空间爆炸的问题。
*   **整体含义**：本文旨在通过一种“实时内核生成”方案，在保证优化效果的同时，极大地缩短编译耗时，使运行时编译在动态模型中真正可行。

### 2. 方法论：核心思想与关键技术
DVM 的核心思想是**基于字节码虚拟机的实时编译**，将传统的“源码 $\rightarrow$ 机器码”流程简化为“源码 $\rightarrow$ 字节码 $\rightarrow$ 虚拟指令执行”。
*   **字节码虚拟机（VM）**：在 CPU 上将算子程序编码为轻量级字节码，在 NPU 上由 Scalar 单元解码为虚拟指令并直接调用预编译的内核函数。
*   **运行时算子编译器**：
    *   **硬件对齐的形状分块（Tiling）**：使用轻量级成本模型和预设模板，根据运行时确定的具体形状进行快速分块，无需硬件实测。
    *   **块级虚拟指令**：定义了 Load/Store 和计算类（如 Add, Matmul, Sum）的块级指令，减少指令数量并对齐 SIMD 架构。
    *   **流水线并行**：利用 NPU 硬件特性，将字节码解码（Scalar 单元）与指令执行（Vector/Cube 单元）流水化，隐藏编译开销。
*   **算子融合器**：
    *   **融合类别**：支持**模式融合**（如 Vector-Vector, Cube-Vector）和**堆叠融合**（空间/时间上的算子堆叠）。
    *   **动态图流式融合**：针对动态图，使用算子缓冲区进行延迟执行，在运行时根据实际路径和形状进行流式融合，无需缓存融合决策。

*   **实验场景**：

    *   **子图级**：LayerNorm, addmm, if-else-add（含动态控制流）。
    *   **模型级**：BERT（训练）、MMoE（推理）、Qwen3-14B（微调）、Llama3.1-8B（微调）。

### 4. 资源与算力
*   **硬件平台**：主要使用华为 **Ascend 910B2** 和 **Ascend 910** NPU。

*   **软件**：CANN 8.5.0, MindSpore 2.7.2/2.8.0, torch-npu 2.7.1。

### 5. 实验数量与充分性
*   **实验规模**：针对算子和子图，随机采样了 **60 种动态形状**进行测试；针对模型，遍历了所有可能的输入形状集。
*   **充分性评价**：实验覆盖了从基础算子到超大规模语言模型（LLM）的完整链路，对比了工业界主流的编译器后端。通过对比运行时间和编译时间两个维度，验证了 DVM 在动态性处理上的全面性。实验设计客观，考虑了 L2 缓存清理等消除干扰的因素。

### 6. 主要结论与发现
*   **执行效率**：DVM 在算子/模型效率上比基准框架提升高达 **11.77 倍**（平均提升 1.09x - 1.82x）。
*   **编译速度**：DVM 的编译耗时比 TorchInductor 快了 **5 个数量级**。在大多数情况下，DVM 的编译开销仅占总运行时间的不到 1%，几乎可以忽略。
*   **动态性支持**：DVM 能够有效处理 `if-else` 等动态控制流中的算子融合，而传统编译器往往在此处发生图断裂或放弃融合。

### 7. 优点
*   **原生支持动态性**：无需预先枚举形状，运行时生成内核，灵活性极高。
*   **极低编译开销**：通过字节码 VM 绕过了耗时的机器码生成和硬件搜索过程。
*   **内存友好**：不依赖庞大的内核缓存，显著降低了设备内存占用。
*   **硬件协同**：充分利用了 Ascend NPU 的多单元并行特性，实现了编译与执行的重叠。

### 8. 不足与局限
*   **硬件依赖性**：虽然设计理念可移植，但目前实现深度绑定在华为 Ascend NPU 架构上，其他加速器（如 GPU）需重新适配 VM。
*   **极小算子开销**：对于计算量极小（如耗时 < 0.007ms）的算子，字节码生成的微小开销仍可能导致其表现略逊于纯 Eager 模式。
*   **融合策略深度**：目前主要支持 Vector 和 Cube 的基础融合模式，对于更复杂的非线性算子融合或跨算子调度优化仍有提升空间。
