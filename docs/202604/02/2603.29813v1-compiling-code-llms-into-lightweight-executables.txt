# Compiling Code LLMs into Lightweight Executables

DAVID LO, Singapore Management University, Singapore The demand for better prediction accuracy and higher execution performance in neural networks continues to grow. The emergence and success of Large Language Models (LLMs) have led to the development of many cloud-based tools for software engineering tasks such as code suggestion. While effective, cloud deployment raises concerns over privacy, latency, and reliance on connectivity. Running LLMs locally on personal devices such as laptops would address these issues by enabling offline use and reducing response time. However, local deployment is challenging: commodity devices lack high-performance accelerators like GPUs and are constrained by limited memory and compute capacity, making it difficult to execute large models efficiently. We present Ditto , a novel method for optimizing both the model size of Code LLMs and their inference programs, particularly for statically-typed programming languages such as C. Our approach integrates two key components: (1) a model compression technique inspired by product quantization, which clusters model parameters into codebooks and quantizes them to lower bit widths while ensuring that outputs remain within a bounded error, as well as synthesizing the inference program for the quantized model; and (2) a compilation pass integrated into LLVM that automatically detects and replaces unoptimized General Matrix-Vector Multiplication (GEMV) operations—the most computationally intensive component in code models—with implementations from Basic Linear Algebra Subprograms (BLAS) libraries, which are highly optimized for runtime performance. The output of Ditto is an optimized and compiled executable for running selected Code LLMs. We evaluate Ditto on three popular Code LLMs—Code Llama, MagicCoder, and OpenCodeInterpreter, achieving up to 10.5 × faster inference, 6.4 × lower memory usage, and 10.5 × reduction in energy consumption compared with their original inference pipeline, while maintaining accuracy close to that of the full-precision models (with an average loss of only 0.27% in pass@1). Furthermore, Ditto outperforms the state-of-the-art int8 quantization baseline, achieving up to 6.96% higher pass@1 accuracy, 2.2 × speedup, and 1.6 × memory usage reduction, which demonstrates the effectiveness of our approach. 1 Introduction 

In recent years, Large Language Models (LLMs) have achieved remarkable success across various domains, particularly specialized LLMs trained on large amounts of source code, which have demonstrated exceptional accuracy in automating tasks such as code generation [ 18 , 30 ]. These models, commonly known as Code LLMs, have become integral to modern software development, enabling developers to write code more efficiently and effectively [ 48 , 77 ]. Their success has led to the creation of various tools and applications, such as GitHub Copilot [ 22 ] and Cursor [ 13 ], which leverage their capabilities to assist developers in software development tasks [66, 77, 79]. While most Code LLM-based applications are deployed on cloud servers to leverage high-performance computing resources, there are notable benefits to running them on personal devices, such as laptops, provided that their memory and computational demands can be reduced to levels manageable by commodity hardware. Recent studies show that developers increasingly prefer local execution of Code LLMs over cloud-based alternatives [ 37 , 45 , 56 , 58 ], as these benefits include improved privacy, lower latency, and offline availability. Running Code LLMs locally allows developers to retain control over their code and data, minimizing the risk of exposing sensitive information to third-party services. In addition, local execution removes the dependency on network connectivity, enabling development in environments with limited or no internet access. It also significantly reduces latency by eliminating data transmission delays to and from cloud servers, which are often affected by network conditions. This is particularly important for real-time applications, such as code completion and debugging, where quick responses are essential. Nonetheless, state-of-the-art Code LLMs typically contain billions of parameters and require 20-30 GB of memory to run, which exceeds the capacity of common laptops or desktops with 16 GB of memory, making direct local deployment impractical without specialized optimization. Currently, only a limited number of studies have investigated reducing the memory and com-putational costs of Code LLMs [ 1 , 57 , 59 , 68 ]. However, these efforts primarily adopt high-level strategies, such as reducing the number or bitwidth of model parameters, without addressing the underlying implementation of model inference from a program optimization perspective. Without low-level optimization, such approaches cannot fully support high-performance execution of Code LLMs on personal devices. For example, although an increasing number of inference pipelines are implemented in C/C++, such as llama2.c [ 34 ], developed by one of OpenAI’s founders to run the LLaMA-series models efficiently, their performance often remains suboptimal because of the lack of systematic low-level optimizations and the frequent reliance on manual tuning for specific hardware. In addition, some implementations depend on specialized high-performance computing hardware, such as GPUs or CPUs with large memory bandwidth, which are not available on most personal devices like business laptops. As a result, their inference programs still require substantial improvement to achieve efficient execution under constrained hardware resources. In this paper, instead of treating model optimization solely as a parameter reduction task, we propose a model quantization method accompanied by a compilation-time optimization approach, framing the compression and acceleration of LLMs as a program optimization problem. We present 

Ditto , a novel tool for jointly optimizing the model size and inference program of Code LLMs, with a current focus on C-based implementations such as llama.c [ 34 ]. As a lightweight and portable pipeline with 18k GitHub stars, llama.c provides efficient LLaMA-series inference on edge devices and has gained wide adoption, serving as a state-of-the-art deployment baseline that our approach

builds upon and further optimizes. Specifically, during inference, an LLM performs computations involving numerous high-precision floating-point parameters. Inspired by the idea of product quantization [ 47 ], which leverages clustering to reduce the bitwidth of model parameters, Ditto 

treats the model parameters as a set of vectors, cluster them, and retain only the centroids of these clusters. The original parameters are replaced with the indices of their corresponding centroids, thereby reducing the bitwidth of the parameters while ensuring that the model’s outputs remain within a specified error bound. After quantization, Ditto automatically synthesizes the inference program for the quantized model. However, simply reducing the bitwidth of model parameters is not sufficient for efficient execution on personal devices. The inference of Code LLMs often involves numerous General Matrix-Vector Multiplication (GEMV) operations, which are computationally intensive and account for over 80% of total execution time [ 38 , 74 ]. To optimize this, we design an LLVM compilation pass that automatically detects GEMV operations and replaces them with highly optimized implementations from Basic Linear Algebra Subprograms (BLAS) libraries, specialized for the target hardware. This combination of quantization and compiler-level optimization yields executables that run Code LLMs efficiently under tight memory budgets while delivering substantial speedups. We evaluated Ditto 

on a personal device equipped with an Apple M2 CPU, which is commonly found in customer-grade laptops. The results show that Ditto enables LLM inference up to 10.5 × faster than the original inference pipeline on the same device, while reducing memory usage by up to 6.4 × and energy consumption by up to 10.5 ×, and maintaining accuracy close to that of the original models, with at most a 4.45% loss in pass@1 and an average loss of only 0.27%. Furthermore, Ditto outperforms an int8 quantization baseline, achieving up to 6.96% higher pass@1 accuracy, up to 2.2 × speedup, 1.6 × lower memory usage, and 2.2 × energy savings. 

Contributions. Our goal in this work is to close the gaps in Code LLM deployment on personal devices that were discussed above. In summary, we make the following contributions: 

• We present Ditto , a novel framework that automatically optimizes the execution of Code LLMs, enabling them to run on personal devices with limited memory and computational resources. 

• We propose a novel model quantization accompanied with a synthesis- and compiler-inspired approach, which reduces the bitwidth of model parameters and accelerates the model inference program while maintaining accuracy. 

• We evaluate Ditto on popular Code LLMs, demonstrating that it can run these models on personal devices with up to 10.5 × faster inference and 6.4 × lower memory usage compared to the original inference programs, while preserving functional correctness with an average loss of only 0.27% in pass@1. 2 Preliminaries 2.1 Code LLMs and Their Inference 

Code LLMs are specialized Large Language Models (LLMs) trained on large amounts of source code [ 18 , 27 , 66 , 75 ], enabling them to perform a variety of software engineering tasks such as code generation [ 10 , 11 , 72 ] and summarization [ 2 , 3 , 39 ], vulnerability detection [ 51 , 62 , 80 ], and program repair [ 31 , 32 , 70 ]. These models are typically based on the Transformer architecture [ 65 ], whose main components are self-attention and feed-forward layers. In the feed-forward layers,

one of the most computationally intensive operations is the General Matrix-Vector Multiplication (GEMV), which is used to compute their outputs. 

General Matrix-Vector Multiplication (GEMV). GEMV is a fundamental linear algebra operation that has regained prominence as a critical building block in neural network inference. In the context of Code LLMs deployed on personal devices—where user queries are typically processed sequentially—GEMV plays a particularly crucial role. During each step of autoregressive generation, the model must multiply large weight matrices by single input vectors. This operation, precisely GEMV, requires streaming entire weight matrices through memory for each token generation, creating significant memory bandwidth pressure. Without careful optimization, inference latency becomes bottlenecked by memory transfers rather than compute. 2.2 Model Compression 

Model compression aims to reduce the size of a model, thereby optimizing other important metrics such as inference latency, memory usage, and energy consumption. Various techniques have been explored for compressing LLMs, including knowledge distillation [ 26 ]. As the pioneering work on compressing Code LLMs, Shi et al. [ 59 ] propose Compressor, which uses a genetic algorithm to simplify models and applies knowledge distillation to reduce LLM size to as little as 3 MB. Their compressed models achieve an average 4.23 × improvement in inference latency. Building on this, Shi et al. [ 57 ] present Avatar, which searches for Pareto-optimal compressed models that jointly optimize model size, inference latency, energy consumption, and carbon footprint, rather than focusing solely on size. Avatar’s compressed models reduce energy consumption by up to 184 ×, carbon footprint by up to 157 ×, and inference latency by up to 76 ×. Additionally, Su and McMillan [ 60 ] apply knowledge distillation to GPT-3.5, producing a smaller and more efficient model for code summarization. Quantization is another widely used approach to compression, which reduces the bitwidth of model parameters. This technique has also been applied to Code LLMs. Wei et al. [ 68 ] and Kaushal et al. [ 36 ] adopt quantization and low-rank decomposition, respectively, to improve efficiency in code generation. In particular, they directly quantize model parameters to 4-bit integers by truncating the least significant bits of the original floating-point values. While simple and effective, this method does not guarantee that the quantized model will produce outputs within a specified error bound. In contrast, Ditto adopts a more sophisticated quantization strategy that clusters model parameters into codebooks and replaces them with lower-bitwidth centroid indices, while ensuring that the model’s outputs remain within a given error bound. 

3 Methodology 3.1 Overview 

Ditto is a two-phase optimization framework that combines weight quantization with program-level optimizations to accelerate inference in Code LLMs. Figure 1 provides an overview of Ditto ’s workflow. The first phase focuses on compressing model weights and synthesizing computation programs, while the second phase optimizes the execution of these programs on the target hardware. In the first phase, Ditto introduces a novel weight quantization method that uses clustering and bit-packing techniques to reduce the memory footprint of model weights. Each layer’s weight matrix is processed by applying clustering to identify representative values (centroids) and replace  

> Proc. ACM Softw. Eng., Vol. 1, No. FSE, Article 1. Publication date: April 2026. 1:6 Shi et al.

the original weights with indices pointing to these centroids, which are then efficiently stored using bit-packing, significantly reducing the overall model size. In the second phase, Ditto applies a customized LLVM IR pass that detects and optimizes General Matrix-Vector Multiplication (GEMV) operations, replacing them with high-performance library calls. This two-phase design enables Ditto to achieve significant reductions in both memory usage and inference latency, while preserving the model’s performance. 

3.2 Weight Quantization 

3.2.1 Clustering. The first aim of Ditto is to represent the weights with few representative float32 values. This amounts to clustering the weights and assigning a representative value per cluster. We use clusters of equal population. Intuitively, this objective will put more clusters where weights are densely populated affording higher resolution where there are more weights. This approach can be implemented by sorting the weights and dividing them to equally sized clusters. The first and the last weight in each cluster determine the boundaries of that cluster. Then the average of the weights inside each cluster can be used as the centroid.

Pseudocode 1: DITTO’s Weight Quantization To reduce this error, Ditto uses an iterative approach that bears sim-ilarity to K-Means [ 23 ]. The process repeatedly applies the following two steps: 1) Ditto moves a weight from cluster A to cluster B if the L1 dis-tance of the weight and the centroid of cluster A is greater than the L1 dis-tance of the weight and the centroid of cluster B. 2) After re-assigning the weights to clusters, Ditto updates the centroids by computing the new average over the weights of each cluster. This is equivalent to minimizing the L2 distance but with a higher convergence rate and speed as studied in prior works [ 73 ]. This iterative process repeats until the sum of L1 distances between centroids and weights is minimized. Algorithm 1 summarizes how Ditto compacts each layer. After the clustering is complete, each weight is replaced with the index of its closest centroid. This index can be stored in a low-bitwidth format, significantly reducing the memory footprint of the model weights. 

Example. For better illustration, Consider a weight matrix of an LLM [0.91, 0.92, 0.89, -0.05, -0.06, -0.04, 1.20, 1.21, 1.19], and suppose we would like to quantize it into 3 clusters. Ditto first sorts the weights: [-0.06, -0.05, -0.04, 0.89, 0.91, 0.92, 1.19, 1.20, 1.21]. Then, it partitions the sorted weights into 3 equal-sized clusters: 

• Cluster 1: [-0.06, -0.05, -0.04] → centroid = -0.05 

• Cluster 2: [0.89, 0.91, 0.92] → centroid = 0.91 

• Cluster 3: [1.19, 1.20, 1.21] → centroid = 1.20 These centroids form the codebook: [-0.05, 0.91, 1.20]. Each original weight is replaced with the index of its closest centroid, resulting in the index sequence [1, 1, 1, 0, 0, 0, 2, 2, 2]. Since only 3 centroids are used, each index can be stored with 2 bits (as ⌈log 2 3⌉ = 2). Compared to the original float32 weights (each occupying 32 bits), using 2-bit indices results in a 16 × reduction in memory for the quantized weights (from 32 bits to 2 bits). The codebook itself requires only 3 × 32 = 96 bits and is shared across all weights in the layer, making its overhead negligible for large layers.

3.2.2 Bit-Packing. The above quantization replaces the vast majority of a model’s high-precision weights with low-precision indices that point to a small, shared dictionary of representative values, i.e., centroids. For instance, if a layer’s weights are quantized using 8 centroids, each weight can be represented by a 3-bit index ( 23 = 8). This allows us to pack the indices into bytes, significantly reducing the model’s memory footprint. However, to realize the full memory savings from such a scheme, these b-bit indices must be stored efficiently. Standard data types are byte-aligned (e.g., an 8-bit char ), so storing a 3-bit index in an 8-bit container would waste over 60% of the storage space. Therefore, a low-level bit-packing mechanism is essential to contiguously serialize these indices into a compact byte stream. The implementation of this mechanism is detailed below and shown in Listing 2 and Listing 3 .

Packing Process. The pack_bits function ( Listing 3 ) is responsible for serializing a sequence of 

b-bit indices into a dense byte array. The core of the algorithm is a running bit_offset counter, which tracks the next available bit position in the output byte stream. For each b-bit index v, its target location is calculated by decomposing bit_offset into a byte index ( j = bit_off » 3 ) and an intra-byte shift ( sh = bit_off & 7 ). The index v is then shifted left by sh positions and merged into the target byte out[j] using a bitwise OR operation ( |= ). This ensures that previously packed bits within the same byte are preserved. A critical aspect of the implementation is its ability to handle indices that span byte boundaries. This occurs when the space remaining in the current byte is insufficient to hold the entire b-bit index (i.e., when sh + b > 8 ). In this case, the algorithm splits the index: the lower-order bits are written to the current byte out[j] , while the higher-order overflow bits are shifted appropriately and written into the beginning of the subsequent byte, out[j+1] . After each index is packed, the 

bit_offset is incremented by b, preparing for the next operation. 

Unpacking Process. Conversely, the unpack_bits function ( Listing 4 ) performs the inverse operation, reconstructing the original b-bit indices from the packed byte stream. The primary challenge here is to efficiently read b bits that may begin at any arbitrary bit position. The function employs a “sliding window” technique to solve this. Instead of reading a single byte, it forms a larger chunk (a 16- or 32-bit integer) by combining the current byte packed[j] and the next byte packed[j+1] . This chunk acts as a lookahead buffer, guaranteeing that the complete 

b-bit representation of any given index is available within this window, regardless of whether it crosses the byte boundary. This approach simplifies the extraction process immensely. The chunk is first right-shifted by 

sh to align the desired index to the least-significant bit position. Then, a bitwise AND operation with a pre-calculated mask isolates and extracts the b-bit index value. This method avoids complex conditional logic and is highly efficient.

3.3 Compilation-Time Optimization 

After having quantized the weights, as well as the inference programs, Ditto applies an optimization pass specifically targeting the above synthesized GEMV operations. This operation is to let the compiler recognize and optimize these patterns effectively and run the GEMV operations using high-performance libraries such as BLAS or OpenBLAS. This section describes how Ditto integrates an LLVM IR pass that rewrites GEMV code and plugs into LLVM’s standard optimization pipeline. 

Ditto provides a custom IR pass that extends LLVM’s PatternMatch API with new matchers designed to recognize GEMV patterns that arise in real-world code. 

Ditto ’s optimization pipeline operates in three key stages: (1) Identify potential GEMV patterns using custom IR matchers; (2) Check the legality of replacing the identified code; (3) Replace the GEMV with a high-performance library call. 

Stage 1: Pattern Identification. Identifying GEMV patterns in LLVM IR is non-trivial due to the diversity of code generation patterns. Although LLVM provides standard PatternMatch APIs, it is insufficient as it does not account for the variations in how GEMV code is formed in practice. Thus, Ditto introduces custom matchers to support variations in loop nesting, induction variable lowering, and memory access patterns.

Stage 2: Legality and Dependency Checks. Once a potential GEMV candidate is matched, it undergoes legality checks to ensure safe transformation. 3.4 Implementation 

Ditto is implemented in 4,000 lines of Python and C++, including code, tests, and documentation. The core functionality is built around llama2.c [ 34 ], a popular open-source implementation of LLM inference in C. Model weights and tokenizers are converted from the Hugging Face model checkpoint format into llama2.c’s native binary layout via its export pipeline, after which Ditto 

operates on the converted binaries. 

4 Evaluation 4.1 Experimental Setup 

Platform. We evaluate Ditto on a machine with an Apple M2 Chips, 24GB of RAM. We use Clang 14 as the compiler to compile the generated C code into an executable binary. The BLAS library used in our experiments is Apple’s Accelerate [ 4 ], which is optimized for Apple Silicon chips. The operating system that we use is macOS 13.2.1. 

Comparisons. We evaluate Ditto by comparing the performance of the optimized programs it generates against two existing implementations as baselines. Note that in the following experiments, we set Ditto to use 3-bit quantization for all models, which is the default setting in Ditto . We leave the exploration of different bitwidths to future work. We first compare Ditto against the original floating-point implementations, llama2.c [ 34 ] which is used in previous work [ 53 , 68 ]. We chose llama2.c as the baseline for several compelling reasons. Firstly, llama2.c, which is developed by the OpenAI’s co-founder Andrej Karpathy, represents the cutting-edge in LLM deployment on edge devices, featuring highly optimized kernel implementa-tions tailored to each hardware platform. Its versatility and robust performance make it an ideal benchmark for assessing the efficacy of new methodologies. Additionally, llama2.c is implemented in plain C/C++ without any dependencies, ensuring maximum compatibility and efficiency across diverse hardware configurations. The llama2.c implementation is with the commit d0f8b3c .Moreover, we also compare Ditto against a strong int8 baseline [ 68 ], a widely adopted quanti-zation method that has demonstrated competitive performance in code generation tasks in prior studies [ 1 , 19 , 68 ]. HumanEval [ 11 ] and MBPP [ 6] are among the most commonly used benchmarks for evaluating code generation. Each task in these benchmarks includes a prompt in the form of a task description (e.g., a docstring), and the LLM is expected to generate code that is evaluated against a set of test cases to verify correctness. The original HumanEval benchmark consists of 164 Python programming problems, while MBPP includes 974 problems. For a more comprehensive evaluation, we use their extended versions, HumanEval+ and MBPP+, which are enhanced by EvalPlus+ [ 42 ]with additional test cases—80 times more for HumanEval and 35 times more for MBPP. The evaluation metrics include pass@1 and pass@5, which measure the percentage of problems for which the generated code passes all test cases on the first and fifth attempts, respectively. We also measure memory usage and inference speed, as these are important non-functional properties for deploying code LLMs on resource-constrained devices.

We evaluate three configurations: (1) Quantization Only , which applies 3-bit weight quantization and the synthesized inference program without LLVM-based GEMV-to-BLAS replacement; (2) BLAS Only , which applies compilation-time GEMV optimization to the full-precision model without quantization; and (3) Full Ditto , which combines both phases. Table 3 summarizes the results. Because BLAS replacement does not modify the model weights, its accuracy matches that of the full-precision baseline. Similarly, Quantization Only and Full Ditto use identical quantized weights and therefore achieve the same accuracy.

The results in Table 3 show that the two phases address distinct bottlenecks and provide comple-mentary benefits. Quantization achieves a 6.4 × memory reduction, compressing the 7B model from 26.77GB to 4.2GB, which allows it to fit within the capacity of commodity laptops. However, it does not improve throughput; performance slightly decreases to 0.8 × of the baseline due to the runtime overhead of bit unpacking and centroid lookup during each GEMV computation. In contrast, BLAS replacement yields a 4.8 × speedup by replacing naive loop-based GEMV with hardware-optimized library calls, but it provides no memory savings because the weights remain in full precision. When both phases are combined, Ditto achieves a 7.0 × speedup, which is substantially higher than the 4.8 × gain from BLAS replacement alone. This improvement arises because quantization reduces the memory bandwidth required for GEMV, allowing the BLAS kernels to operate more efficiently and enabling greater acceleration than either technique alone. These results confirm that quantization and compilation-time optimization are synergistic rather than independent: quantization alleviates the memory bandwidth bottleneck, BLAS replacement mitigates the compute bottleneck, and their integration yields amplified improvements in throughput, memory usage, and energy efficiency. 

Answer to RQ3: Ditto ’s two phases address complementary bottlenecks. Together, they achieve a 7.0 × speedup, a 6.4 × memory reduction, and a 7.1 × reduction in energy consump-tion, outperforming either quantization or BLAS replacement alone due to their synergistic interaction. 

5 Additional Analysis 5.1 Time Cost of Quantization and Compilation 

We measure the quantization and compilation time required by Ditto to generate an optimized executable from a 7B-parameter Code LLM. On our evaluation machine (Apple M2 CPU, 24GB RAM), the complete pipeline, including model quantization, bit-packing, and LLVM-based GEMV optimization, completes in approximately 17 minutes for all three models. This processing time is relatively short compared to the overall lifecycle of deploying a Code LLM, as it represents a one-time cost per model configuration. Once the optimized binary is generated, it can be reused indefinitely without recompilation unless the model weights or target hardware change. The 17-minute duration highlights the practicality of Ditto in real-world scenarios, allowing developers to efficiently convert large models into executables suitable for on-device inference. Moreover, given the sustained performance improvements during inference—up to 10.5 ×

faster execution and 6.4 × lower memory usage as demonstrated in RQ2—the time overhead is minimal and can be easily amortized over multiple inference runs.

More broadly, this comparison highlights a fundamental architectural distinction. GPU-oriented methods such as AWQ [ 41 ] and GPTQ [ 20 ] depend on specialized CUDA kernels for low-bit arithmetic and revert to higher-precision computation on CPUs, which diminishes their theoretical gains in CPU settings. Meanwhile, classical Vector Quantization (VQ) [ 54 ] and its variant, Product Quantization (PQ) [ 47 ], share a similar high-level idea with our method, i.e., reducing bitwidth through clustering. They quantize weight blocks into codebooks and perform inference through lookup-table accumulation. Although memory efficient, these approaches introduce additional overhead from codebook access and reconstruction. Without optimizing the inference program, this overhead can offset the benefits of compression on CPUs, as shown in our RQ3 ablation study (Section 4.4). In contrast, our approach not only compresses weights but also optimizes the inference program. By replacing costly GEMV loops with BLAS-optimized kernels, Ditto compensates for the overhead of codebook lookups and enables efficient execution of low-bit models on commodity CPUs. This co-design of quantization and program-level optimization provides a technically distinct perspective on efficient LLM inference under tight resource constraints. 

5.4 Generalization of Ditto 

A natural question is whether the optimizations in Ditto are specific to the LLaMA architecture used in our primary evaluation. To assess generality, we apply Ditto to models with different architectures, scales, and domains (Table 6). 

Architecture Generality. Both components of Ditto , namely weight quantization and GEMV-to-BLAS replacement, operate on standard Transformer linear layers and do not depend on architecture-specific mechanisms. We validate this on CodeQwen-1.5-7B [ 52 ], which adopts the Qwen archi-tecture [ 7 ] and differs from LLaMA in its attention mechanism and feed-forward network design. Applying Ditto to CodeQwen-1.5-7B yields a memory reduction of 6.1 ×, reducing memory us-age from 27.1GB to 4.4GB, and achieves a 3.9 × speedup, which is comparable to the results for Code Llama-7B. These results confirm that Ditto ’s optimizations are not tied to a specific model architecture and can be effectively applied to other Transformer-based LLMs. 

Scale and Domain Generality. We further evaluate Ditto on Code Llama-13B [ 55 ] and TinyLLaMA-110M [ 35 ] to examine scalability across model sizes and domains. For Code Llama-13B, memory usage decreases from 53.0GB to 9.7GB (5.5 ×) with a 6.1 × speedup. Notably, the full-precision 13B model exceeds the 24GB memory capacity of our evaluation machine, whereas the quantized version fits comfortably, demonstrating that Ditto enables practical deployment at larger scales.

For TinyLLaMA-110M, which is a general-purpose language model rather than a code-specialized one, memory usage decreases from 0.45GB to 0.11GB (4.1 ×) with a 4.2 × speedup. Hardware Portability. Although our evaluation is conducted on an Apple M2 CPU, Ditto is not platform-specific. The GEMV-to-BLAS replacement is implemented as an LLVM IR pass that emits calls to the standard CBLAS interface. At link time, these calls bind to the local BLAS provider, such as Apple Accelerate, OpenBLAS, or Intel MKL. Since optimized BLAS libraries are available for modern CPU architectures, including x86, ARM, and RISC-V, the same compiled pipeline can be deployed by relinking against the target platform’s BLAS implementation. Memory reduction remains consistent across platforms, while throughput improvement depends on the quality of the underlying BLAS library. 

5.5 Implications for Software Engineering 

Takeaway 1 (for software developers): With Ditto , developers can run code LLMs locally on standard laptops without GPUs, achieving latency below 300ms with approximately 4GB of memory. This enables fast, privacy-preserving code completion and analysis directly within IDEs, without transmitting proprietary code to cloud services. 

Takeaway 2 (for engineering teams): Teams can integrate Ditto as a build-time step that converts pretrained models into compact, GPU-free C executables in approximately 17 minutes (Section 5.1). The resulting binary depends only on a system BLAS library, which facilitates seamless integration into CI/CD pipelines and supports on-device deployment of AI coding assistants. 

Takeaway 3 (for SE researchers): Ditto illustrates that co-designing model compression with program-level optimization yields synergistic gains that exceed those of either technique alone (Sec-tion 4.4). This principle can guide future research on efficient LLM-powered software engineering tools, such as on-device program repair and privacy-sensitive code search. 

5.6 Threats to Validity 

A potential threat to internal validity arises from randomness in code generation and quantization. To mitigate this issue, we repeated each experiment five times with different random seeds and reported the average results, following Arcuri and Briand [ 5 ]. Although clustering in quantization depends on centroid initialization, our iterative refinement process converges to a stable configura-tion, which reduces sensitivity to randomness. Another concern is the correctness of our custom LLVM pass, which we addressed through extensive unit testing and by verifying functional equiva-lence between the original and optimized code on representative examples. Regarding external validity, we evaluated three widely used code-specific LLMs on two standard benchmarks using an Apple M2 CPU. These models, datasets, and hardware are representative of common usage scenarios for code LLMs. Nevertheless, future work could further examine generalizability across additional models, benchmarks, and platforms. For construct validity, our metrics, including pass@1, pass@5, inference speed, and memory usage, are standard in LLM evaluation and directly reflect practical usability. We also compared our approach with a strong INT8 quantization baseline. Therefore, we believe that the threat to construct validity is limited.

6 Related Work 

The first line of related work focuses on model compression [ 44 , 58 ], which reduces the size of LLMs to improve inference latency, memory usage, and energy consumption. As an early effort on com-pressing code LLMs, Shi et al. [ 59 ] use a genetic algorithm to simplify model architectures and apply knowledge distillation to shrink models, achieving an average 4.23 × latency improvement, and subsequently introduce Avatar [ 57 ], which jointly optimizes model size, latency, energy consump-tion, and carbon footprint. Similarly, Su and McMillan [ 60 ] distill GPT-3.5 into a smaller model for code summarization. Other approaches include quantization [ 68 ] and low-rank decomposition [ 36 ], both of which achieve notable efficiency gains for code generation. Recent work further examines the non-functional properties of compressed code LLMs, including robustness [ 19 ], privacy [ 24 ], and output code quality [1]. Moreover, there have been efforts on software-hardware co-design for efficient LLM inference [ 49 ,50 , 67 , 71 ], which jointly optimize model and underlying hardware architectures to improve performance. There are also studies designed for high-performance platforms such as GPUs or ASIC accelerators, including VQ-LLM [ 43 ], MVQ [ 40 ], and VQT-CiM [ 28 ]. VQ-LLM depends on custom CUDA kernels, MVQ relies on GPU tensor cores, and VQT-CiM targets ferroelectric compute-in-memory accelerators. These hardware-specific implementations differ from our focus on commodity CPU-only devices without such hardware support. In addition, neural codebook methods such as QINCo2 [ 64 ] address a fundamentally different problem: compressing input vectors into quantized embeddings for similarity search rather than compressing model weights for LLM inference. Also, such methods use neural networks to construct codebooks and reconstruct vectors, whereas Ditto performs table lookup to recover weights and co-designs quantization with compilation-time GEMV-to-BLAS replacement, which enables low-bitwidth execution while systematically optimizing the inference program itself, an integration that has not been explored in prior work. We therefore believe that our method is both technically novel and provides a unique perspective in this field. Program-level optimization of code LLMs has also been explored, primarily by restructuring inputs or reducing token counts to lower processing costs, since inference time scales with input length. Token pruning methods remove less important tokens to shorten sequences [ 25 , 61 , 76 ]. For example, Zhang et al. [ 76 ] propose DietCode, which selects high-attention statements and tokens to reduce the computational cost of Code LLMs. Although effective in shrinking inputs, these approaches do not optimize the underlying inference program itself, which limits overall system-level gains. In contrast, our work directly improves the low-level execution of the inference pipeline and complements input-level reductions with compiler-based acceleration. Beyond LLM-specific efforts, compiler research has long leveraged pattern matching and LLVM IR passes for program optimization, such as LoopTactics [ 9 ], KernelFaRer [ 15 ], and MMTR [ 29 ], which perform nested loop and matrix multiplication optimizations through LLVM-based idiom recognition and rewriting. However, these systems target general-purpose program optimization rather than LLM inference workloads. Our work instead introduces a specialized framework tailored to code LLM inference, providing a focused and high-performance solution for this emerging domain. 

7 Conclusion and Future Work 

In this work, we present Ditto , a synthesis- and compiler-inspired framework for optimizing code LLM inference programs. By combining quantization with LLVM-based GEMV optimization, 

Ditto generates low-bitwidth, memory- and compute-efficient executables that run on commodity CPUs. When evaluated on Code Llama, MagicCoder, and OpenCodeInterpreter, Ditto achieves up to 10.5 × speedup and 6.4 × memory reduction over the original pipelines while maintaining near full-precision accuracy, with an average 0.27% pass@1 loss.
