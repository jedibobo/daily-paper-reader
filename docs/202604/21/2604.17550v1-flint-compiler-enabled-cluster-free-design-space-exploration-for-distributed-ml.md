## 摘要
未来分布式机器学习系统的设计空间探索面临着缺乏现成负载表示的问题，而这种表示对于实现跨技术栈的灵活探索至关重要。我们提出了 Flint，这是一个利用机器学习框架编译器的中间表示（Intermediate Representation）来填补这一空白的框架。编译器承担了理解并保留原始模型代码行为的核心重任。由于 Flint 在硬件执行前与编译器对接，因此它可以收集任意集群规模的负载表示。我们通过执行后的追踪记录验证了负载图的准确性，并通过一个设计空间探索案例研究展示了 Flint 的灵活性。

## Abstract
Design space exploration for future distributed Machine Learning systems suffers from a lack of readily available workload representation that enables flexible exploration across the stack. We present Flint, a framework that bridges this gap by leveraging the Intermediate Representation of Machine Learning framework compilers. The compiler does the heavy weight lifting of understanding and preserving the behavior of the original model code. Flint can collect the workload representation of arbitrary cluster size because it interfaces with the compiler before hardware execution. We validate the workload graph against post-execution traces and show the flexibility of Flint through a design space exploration case study.

#### 1. 核心问题与研究动机
分布式机器学习（ML）系统的设计空间探索（DSE）面临“两难”境地：
*   **现有方法的局限：** 在真实集群上运行（In-stack execution）成本极高且受限于现有硬件；基于模拟器的方法虽然灵活，但获取准确的“工作负载表示”（Workload Representation）非常困难。
*   **负载获取的痛点：** 传统的合成生成器（Synthetic Generation）难以跟上快速演进的模型架构和并行策略；而基于执行后追踪（Post-execution Traces）的方法要求用户必须先拥有同等规模的 GPU 集群，这违背了模拟探索的初衷。
*   **核心目标：** Flint 旨在提供一种**无集群（Cluster-Free）**的方法，通过机器学习编译器直接提取高保真的工作负载图，从而在无需实际 GPU 的情况下，支持对未来大规模系统和新型硬件的探索。

#### 2. 方法论
Flint 的核心思想是利用 **PyTorch 编译器的中间表示（IR）** 来捕获模型行为。
*   **关键技术细节：**
    *   **编译器钩子（Compiler Hook）：** Flint 作为一个自定义后端注册到 PyTorch 的 `torch.compile` 中。它在 **AOT Autograd** 阶段之后捕获 FX Graph。这一阶段的 IR 已经包含了前向和反向传播，且保留了真实的**数据依赖关系**，而非简单的 API 调用顺序。
    *   **GPU 运行时幻觉（Illusion of GPU Runtime）：** 为了在无 GPU 环境下运行代码，Flint 强制模型在 **Meta Device**（元设备）上初始化。Meta Device 仅记录张量的形状和元数据，不分配实际显存，从而允许在单台 CPU 机器上模拟成千上万个 GPU 节点的行为。
    *   **图转换与标准化：** Flint 将捕获的 FX Graph 转换为 **Chakra Graph**（一种由 MLCommons 维护的标准化执行追踪格式）。这使得 Flint 生成的负载图可以无缝对接 ASTRA-sim（模拟器）、Genie（网络仿真器）等多种下游工具。
    *   **调度灵活性：** 由于捕获的是原始数据依赖，Flint 允许用户在模拟阶段探索不同的算子调度策略（如通信重排序），而不会被特定的硬件实现锁死。

#### 6. 主要结论与发现
*   **高保真度：** Flint 生成的预执行图在算子分布和端到端性能预测上与真实执行追踪高度一致。
*   **设计空间洞察：** 实验发现通信重排序在不同带宽下效果迥异：高带宽下可提升 7% 性能，但在低带宽下由于通信占主导，重排序收益微乎其微。
*   **无集群优势：** 证明了可以在没有 GPU 的情况下，通过 CPU 节点准确预测大规模分布式训练的行为，极大地降低了研究门槛。

#### 7. 优点
*   **解耦硬件与软件：** 实现了在没有物理集群的情况下进行大规模分布式系统的 DSE。
*   **捕获真实依赖：** 相比于 CUDA API 拦截方法，基于编译器 IR 的方法能捕获真正的算子间数据流，支持更深度的调度优化探索。
*   **生态兼容性：** 采用 Chakra 标准格式，能够直接利用现有的模拟和分析工具链。
*   **易用性：** 对原始 PyTorch 代码的修改极小（仅需指定设备为 Meta）。

#### 8. 不足与局限
*   **框架依赖：** 目前主要针对 PyTorch 及其 FX Graph，虽然理论上可扩展至 JAX (Jaxpr)，但尚未实现。
*   **动态性限制：** `torch.compile` 在处理包含复杂数据依赖控制流（Graph Breaks）的代码时仍有局限，这可能导致生成的图不完整。
*   **算子性能模型：** Flint 依赖离线分析或第三方模型来提供算子的执行时长。如果目标硬件是全新的，仍需要准确的算子性能预测模型支撑。
*   **低级细节缺失：** 编译器 IR 无法完全捕获某些硬件底层的自动分解行为（如特定算子在 CUDA 内部的拆分），这在验证实验中导致了少量杂项算子的计数偏差。
