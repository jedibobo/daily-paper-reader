## 摘要
随着模型权重和 KV 缓存的快速增长，大语言模型（LLM）推理日益受到 GPU 中高带宽内存（HBM）容量的限制。高带宽闪存（HBF）在保持相当带宽的同时，提供了比 HBM 更高的容量，使其成为受容量限制的 LLM 推理的一种极具前景的底层介质。然而，其固有的高访问延迟、低带宽利用率以及缺乏对异构资源管理的支持，使得将 HBF 集成到 GPU 中进行 LLM 推理变得十分困难。我们提出了 FlashAccel，这是一个协同设计的系统，能够利用 HBF 实现高效的 LLM 推理。FlashAccel 将 HBF 集成到基于 HBM 的 GPU 中，提供架构支持以缓解访问延迟。它通过针对模型权重和 KV 缓存的专门数据布局提高了带宽利用率，并引入了一个感知 HBF 的存储管理层以及一种编程模型，用于在 HBF 中组织持久化数据并在系统层面协调异构内存资源。实验结果表明，在 100ms 延迟约束下，将六个 HBF 堆栈集成到 GPU 中，使 FlashAccel 在单 GPU 吞吐量和能效方面，相较于仅使用 HBM 的 GPU 分别实现了平均 2.54 倍和 1.93 倍的提升。

## Abstract
Large language model (LLM) inference is increasingly limited by the capacity of High-Bandwidth Memory (HBM) in GPUs, as model weights and KV cache grow rapidly. High-Bandwidth Flash (HBF) provides higher capacity than HBM while retaining comparable bandwidth, making it a promising substrate for capacity-constrained LLM inference. However, its inherently high access latency, low bandwidth utilization, and lack of support for heterogeneous resource management make it difficult to integrate HBF into GPUs for LLM inference. We present FlashAccel, a co-designed system that enables efficient LLM inference using HBF. FlashAccel integrates HBF into HBM-based GPUs, providing architectural support to mitigate access latency. It improves bandwidth utilization through specialized data layouts for both model weights and KV cache, and introduces an HBF-aware storage management layer together with a programming model to organize persistent data in HBF and coordinate heterogeneous memory resources at the system level. Experimental results demonstrate that integrating six HBF stacks into the GPU enables FlashAccel to deliver an average improvement of 2.54$\times$ and 1.93$\times$ in throughput per GPU and energy efficiency over the HBM-only GPU under 100ms latency constraint, respectively.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：大语言模型（LLM）推理正面临严重的**内存容量瓶颈**。随着模型参数量（Weights）和上下文长度（KV Cache）的激增，GPU 昂贵且容量有限的 HBM（高带宽内存）已无法满足需求，导致推理时 Batch Size 受限（吞吐量低）、KV 缓存频繁换出（重计算开销大）以及多 GPU 部署成本高昂。
*   **研究背景**：高带宽闪存（HBF）虽然提供了与 HBM 相当的带宽和远超其的容量，但存在**高访问延迟**、**低带宽利用率**（需极高并行度）以及**异构资源管理困难**等挑战。本文旨在通过软硬件协同设计，使 HBF 能够高效集成到 GPU 推理流程中。

### 2. 方法论：核心思想与关键技术
FlashAccel 采用了从硬件架构到系统软件的全面协同设计：
*   **硬件架构设计**：
    *   **HBF 集成**：提出了 CLI（共置集成）和 CSI（级联集成）两种方案，将 HBF 堆栈与 GPU 连接。
    *   **SRAM 预取缓冲**：在 HBF 的电路层和基础层集成 SRAM 缓存，用于实现数据预取流水线，隐藏闪存的毫秒级访问延迟。
*   **数据布局优化**：
    *   **权重布局**：根据推理执行顺序，将模型权重以 Page 为单位轮询（Round-robin）分布在所有闪存平面（Planes）上，确保访问时的负载均衡。
    *   **KV 缓存布局**：引入“统一 KV 缓存”抽象，通过**超页（Hyper-page）写入**平衡平面负载；针对运行时的负载倾斜，采用**选择性卸载（Selective Offloading）**将过载平面的数据迁移至 HBM。
*   **系统与编程模型**：
    *   **HBF 感知存储层**：放弃复杂的闪存转换层（FTL），利用 LLM 数据的追加写特性，直接维护物理地址映射，降低元数据开销。
    *   **异步预取编程接口**：提供 `SramPrefetch` 等接口，允许系统在计算当前层时，异步将下一层数据从 HBF 加载至 SRAM，实现计算与传输的重叠。

*   **基准测试（Benchmark）**：以 NVIDIA DGX-H200 节点（8×H200 GPU）为基准。
*   **对比方法**：对比了纯 HBM GPU、CLI 架构、CSI 架构，以及在不同 GPU 数量（4/8/16）下的表现。

### 5. 实验数量与充分性
*   **实验规模**：进行了多维度的评估，包括不同模型下的吞吐量对比、不同延迟约束（50ms/100ms SLO）下的表现、消融实验（验证预取和布局优化的有效性）、能效分析、面积开销分析以及闪存寿命（Endurance）分析。
*   **充分性评价**：实验设计较为全面，考虑了长文本和多轮对话等前沿需求，消融实验清晰地展示了各组件对性能的贡献。通过模拟不同规模的 GPU 集群，证明了 HBF 在减少跨节点通信方面的潜力，实验结果具有较强的说服力。

### 6. 主要结论与发现
*   **吞吐量提升**：在 100ms 延迟约束下，FlashAccel 相比纯 HBM 系统，单 GPU 吞吐量平均提升 **2.54 倍**。
*   **能效优化**：得益于吞吐量的巨大提升，尽管闪存读取功耗高于 HBM，但整体能效（Tokens/J）提升了 **1.93 倍**。
*   **容量红利**：HBF 使得单 GPU 能承载更大的 Batch Size 和更多的 KV 缓存，显著提高了多轮对话中的缓存命中率，减少了重计算。
*   **寿命可行性**：分析表明，使用 SLC 闪存配合 KV 缓存的追加写模式，其寿命足以支撑 5 年以上的典型推理负载。

### 7. 优点
*   **软硬协同深度高**：不仅改进了硬件连接方式，还针对 LLM 的计算图特性优化了底层存储布局。
*   **解决核心痛点**：直接针对 KV Cache 这一当前 LLM 推理中最耗内存的组件进行了优化，而非仅仅关注静态权重。
*   **延迟隐藏机制有效**：通过 SRAM 预取和异步调度，成功弥补了闪存与 HBM 之间的延迟鸿沟。

### 8. 不足与局限
*   **依赖模拟器**：目前所有结论均基于模拟数据，尚未有真实的物理芯片原型验证，实际流片后的热管理和信号完整性可能存在挑战。
*   **SLC 成本问题**：为了保证寿命和速度，方案依赖 SLC 闪存，这在一定程度上削弱了闪存相对于 HBM 的成本优势（虽然仍比 HBM 便宜）。
*   **特定模式依赖**：存储管理层的简化依赖于 LLM 的追加写（Append-only）模式，对于需要频繁随机修改数据的其他 AI 任务可能不适用。
